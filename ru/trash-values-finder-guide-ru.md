# trash-values-finder-guide.md

> Руководство для ИИ: поиск мёртвого кода, оборванных цепей и артефактов рефакторинга в CSS / JS / TS проектах.
> Цель — не просто найти «неиспользуемое», а восстановить логическую цепочку: откуда пришло, куда шло, почему осталось.

---

## Философия: думать цепями, не файлами

Любая сущность в коде — это узел в логической цепи:

```
объявление → передача → использование → эффект
```

Мёртвый код — это цепь с обрывом. Артефакт рефакторинга — цепь, которую переименовали с одного конца, но забыли с другого. Задача ИИ: найти обрыв, понять на каком конце он находится, и принять решение — починить или удалить.

**Три исхода для каждой находки:**
1. **Переименовать** — имя изменилось в одном месте, не изменилось в другом
2. **Удалить** — сущность реально ничего не делает, нет входящих ссылок
3. **Оставить** — находка ложно-положительная (динамика, внешний API, будущее)

## Часть 1. CSS — классы и селекторы

### 1.1 Алгоритм поиска мёртвых классов

**Шаг 1. Собрать все объявленные классы из CSS/SCSS/Less**

Парсить с учётом вложенности (SCSS `&-modifier`), псевдоклассов (`:hover`, `::before`), медиазапросов. Нужны только «корневые» имена классов — без псевдо и комбинаторов.

```
Паттерн: /\.([a-zA-Z_-][a-zA-Z0-9_-]*)/g
```

Исключать из списка «объявленных»:
- Классы внутри `@keyframes`
- Классы в комментариях
- Vendor-prefixed системные селекторы (`:root`, `*`, `html`, `body`)

**Шаг 2. Собрать все использования классов из шаблонов/JS**

Источники использований — смотреть ВСЕ:

| Источник | Паттерн для поиска |
|---|---|
| HTML атрибут | `class="..."`, `className="..."` |
| JSX динамика | `className={styles.foo}`, `` className={`btn ${mod}`} `` |
| JS DOM API | `classList.add/remove/toggle/replace/contains(...)` |
| JS строки | `querySelector('.foo')`, `getElementsByClassName('foo')` |
| CSS Modules | `styles.foo`, `styles['foo-bar']` |
| Template literals | `` `icon icon-${name}` `` → флаг: динамическое, не трогать |
| Data-атрибуты-триггеры | `[data-class="foo"]` в JS |
| Тесты | `getByRole`, `getByTestId`, `.toHaveClass('foo')` |

**Шаг 3. Построить матрицу**

```
для каждого класса из CSS:
  если нет ни одного совпадения в шаблонах/JS → кандидат на удаление
  если совпадение только в комментарии → кандидат на удаление
  если совпадение в template literal → ФЛАГ: динамический, пропустить
```

**Шаг 4. Проверить перед удалением**

Перед удалением класса задать вопросы:
- Это utility-класс (`.mt-4`, `.hidden`, `.sr-only`)? → вероятно нужен, искать в HTML
- Это CSS-only состояние (`.is-open`, `.active`)? → возможно тоггл через JS `.className =`
- Это класс из дизайн-системы / сторонней библиотеки? → не трогать
- Есть ли похожее имя с опечаткой (`.buttton`, `.primray`)? → предложить переименование

---

### 1.2 Артефакты переименования классов

**Симптом:** старое имя в CSS, новое имя в шаблоне (или наоборот).

**Алгоритм обнаружения:**

1. Найти класс в JS/HTML которого нет в CSS → кандидат «используется, но не объявлен»
2. Найти класс в CSS которого нет в JS/HTML → кандидат «объявлен, но не используется»
3. Применить Levenshtein distance или смысловое сравнение между двумя списками:
   - `.btn-primary` (CSS) ↔ `.buttonPrimary` (JS) — артефакт переименования
   - `.card__header` (CSS) ↔ `.card-header` (JS) — смена соглашения BEM → kebab
   - `.isActive` (CSS) ↔ `.is-active` (JS) — camelCase → kebab

**Правила переименования:**
- Если в CSS один файл, в JS много мест → переименовать CSS
- Если CSS генерируется (CSS Modules, Tailwind JIT) → переименовать JS
- Всегда переименовывать атомарно: найти все вхождения через `grep -rn "имя" src/`

### 1.3 CSS-переменные (custom properties)

**Сбор объявлений:**
```
Паттерн: /--([\w-]+)\s*:/g  →  в :root, :host, любых блоках
```

**Сбор использований:**
```
Паттерн 1 (CSS): /var\(--([\w-]+)/g
Паттерн 2 (JS get): /getPropertyValue\(['"`]--([\w-]+)/g
Паттерн 3 (JS set): /setProperty\(['"`]--([\w-]+)/g
Паттерн 4 (JS read): /style\.--([\w-]+)/g  ← редко, но бывает
```

**Три типа проблем:**

```
A. Объявлена, не используется:
   :root { --old-brand-color: #ff0000; }  ← никто не вызывает
   → Удалить

B. Используется, не объявлена:
   color: var(--button-hover-color);  ← нет в :root
   → Найти откуда должна была прийти (переименование?) или добавить

C. Переопределена, но переопределение игнорируется:
   :root { --gap: 1rem; }
   .card { --gap: 2rem; color: var(--gap); }  ← тут --gap = 2rem
   .sidebar { color: var(--gap); }  ← тут --gap = 1rem (корректно, но проверить умысел)
```

**Особый случай — каскадное переопределение:**
Переменная «используется», но её переопределение в компоненте никем не читается → мёртвое переопределение.

## Часть 2. JavaScript / TypeScript

### 2.1 Мёртвые переменные и константы

**Уровень 1 — локальные переменные**

```javascript
// Объявлена, присвоена, не прочитана:
const result = fetchData();  // result нигде ниже не встречается

// Присвоена повторно без чтения первого значения:
let config = defaultConfig;
config = userConfig;  // первое присвоение мёртвое
```

Алгоритм:
1. Найти все `const/let/var` объявления в скоупе
2. Для каждого — найти все чтения (не присвоения) в том же скоупе
3. Если чтений нет → мёртвая переменная

**Уровень 2 — экспортируемые константы**

```typescript
// файл constants.ts
export const API_TIMEOUT = 5000;      // используется
export const OLD_API_URL = '/api/v1'; // нигде не импортируется
export const MAX_RETRIES = 3;         // используется
```

Алгоритм:
1. Собрать все `export const/function/class` из файла
2. Grep по всему проекту: `import { OLD_API_URL }` или `constants.OLD_API_URL`
3. Нет совпадений → мёртвый экспорт

**Исключения — не удалять без дополнительной проверки:**
- `export default` — может использоваться через динамический импорт
- Re-exports (`export { x } from './x'`) — может быть публичный API
- Константы в `index.ts` / `public-api.ts` — это библиотечный интерфейс
- Переменные с префиксом `_` — явно помечены как intentionally unused

---

### 2.2 Мёртвые функции

**Паттерны поиска:**

```
1. Объявлена, не вызывается:
   function formatDate(d) { ... }  // нет formatDate( нигде в проекте

2. Вызывается, но результат игнорируется всегда:
   validateInput(data);  // функция возвращает boolean, который нигде не читается
   → не мёртвая, но подозрительная — возможно эффект-функция, возможно баг

3. Функция-обёртка без добавленной ценности:
   const getUser = (id) => fetchUser(id);  // просто проксирует
   → кандидат на удаление + замену всех вызовов на прямой fetchUser(id)

4. Overloaded / дублирующая:
   function parseDate(s) { ... }  // 40 строк
   function parseDateString(s) { ... }  // 38 строк, делает то же самое
   → найти, сравнить, оставить одну
```

**Алгоритм для мёртвых функций:**
1. Собрать все объявления функций (включая arrow functions в переменных)
2. Для каждой — искать `имяФункции(` по всему проекту (не в объявлении)
3. Учитывать передачу как аргумент: `arr.map(formatDate)` → живая
4. Учитывать object spread: `{ ...helpers, formatDate }` → живая
5. Проверить тесты: `describe('formatDate', ...)` → живая (тестируется)

---

### 2.3 Мёртвые импорты

**Простой случай:**
```javascript
import { formatDate, parseDate } from './utils';
// parseDate нигде не используется в файле
```

**Сложный случай — side-effect импорты:**
```javascript
import './polyfills';           // side-effect, не трогать
import 'reflect-metadata';      // side-effect для декораторов, не трогать
import styles from './Card.module.css';  // используется через styles.xxx
```

**Алгоритм:**
1. Для каждого именованного импорта → поиск использований в теле файла
2. Для default-импортов → аналогично
3. Side-effect импорты (без `{...}` и без `x from`) → всегда помечать как «проверить вручную», не удалять автоматически
4. Импорты типов (`import type { Foo }`) → мёртвые если тип нигде не аннотирован

---

### 2.4 TypeScript — мёртвые типы и интерфейсы

```typescript
// Объявлен, не используется:
interface OldUserShape {
  id: number;
  name: string;
}

// Используется только внутри файла где объявлен — окей
// Экспортируется, но нигде не импортируется — кандидат на удаление

// Частичный артефакт — поля которых нет в реальных объектах:
interface User {
  id: number;
  name: string;
  legacyToken?: string;  // опциональное, никогда не присваивается
}
```

**Алгоритм для полей интерфейса:**
1. Найти все места где тип `User` используется
2. Найти все присвоения `legacyToken` → если ноль → мёртвое поле
3. Найти все чтения `user.legacyToken` → если ноль → подтверждение

---

### 2.5 Мёртвые ветки условий

```javascript
// Константное условие:
const DEBUG = false;
if (DEBUG) {
  console.log('...');  // никогда не выполнится
}

// Условие которое всегда true по типам:
function process(items: string[]) {
  if (items === null) { ... }  // string[] не может быть null в TS strict
}

// Недостижимый default:
type Status = 'active' | 'inactive';
switch (status) {
  case 'active': ...
  case 'inactive': ...
  default:
    throw new Error('Unknown');  // никогда не достигается если тип исчерпан
}
```

**Алгоритм:**
1. Найти `if (CONSTANT_NAME)` где константа явно `true` или `false`
2. Найти сравнения с типами которые TypeScript гарантирует невозможными
3. Отметить `default` в switch как «проверить» если union type исчерпан

## Часть 3. Логические цепи — сквозной анализ

### 3.1 Что такое логическая цепь

Это полный путь данных или зависимости от источника до эффекта:

```
Пример 1 — prop-drilling chain:
  API response → parseUser() → User type → UserCard props → avatar prop → <img src>

Пример 2 — CSS chain:
  design token (--color-brand) → button.css → .btn-primary → <button class="btn-primary"> → видимый цвет

Пример 3 — event chain:
  click → handleClick() → dispatch(updateUser) → reducer → state.user → re-render
```

**Обрыв цепи** — это место где данные «падают»: присваиваются но не читаются дальше, или читаются из источника которого нет.

---

### 3.2 Алгоритм восстановления цепи

Для любой подозрительной сущности (класс, переменная, функция):

```
1. ОТКУДА она приходит?
   - Объявлена локально
   - Импортирована
   - Пришла как prop / аргумент
   - Получена из API / store

2. КУДА она идёт?
   - Передаётся дальше (в функцию, компонент, DOM)
   - Возвращается из функции
   - Присваивается другой переменной
   - Используется в условии
   - Рендерится / применяется

3. ЕСТЬ ЛИ ЭФФЕКТ?
   - Изменяет DOM / стили
   - Отправляет запрос
   - Изменяет состояние
   - Логирует / метрики
   - Возвращает значение которое кто-то использует
```

**Вывод:**
- Если ответы на 2 и 3 — «нигде» и «нет» → удалить
- Если ответ на 1 — «ниоткуда» но 2 и 3 есть → найти источник (артефакт переименования)
- Если цепь есть но запутана → рефакторинг имён, не удаление

---

### 3.3 Специфика после рефакторинга ИИ

ИИ-рефакторинг создаёт характерные артефакты. Знать их паттерны:

**Паттерн 1: Переименование с одного конца**
```css
/* было */ .card-wrapper { ... }
/* ИИ переименовал в JS */ → className="cardContainer"
/* CSS остался */ .card-wrapper { ... }  ← мёртвый
/* CSS не создан */ .cardContainer { ... }  ← отсутствует
```

**Паттерн 2: Инлайнинг функции без удаления оригинала**
```javascript
// ИИ заинлайнил логику parseDate в 3 местах
// но оригинальная функция parseDate осталась объявленной
function parseDate(s) { ... }  // мёртвая
```

**Паттерн 3: Новая переменная-обёртка поверх старой**
```javascript
const userData = user;           // старое имя
const currentUser = userData;    // ИИ добавил
const activeUser = currentUser;  // ИИ добавил ещё раз
// везде дальше используется activeUser
// userData и currentUser — мёртвые промежуточные
```

**Паттерн 4: Дублирование типа**
```typescript
// было
type ApiUser = { id: string; name: string; }
// ИИ создал новый
interface User { id: string; name: string; }
// оба существуют, используется только User
// ApiUser — мёртвый тип
```

**Паттерн 5: Комментарий-зомби**
```javascript
// TODO: remove after migration to v2
const legacyAdapter = ...  // migration давно прошла, легаси живёт
```

---

### 3.4 Матрица решений

| Ситуация | Действие |
|---|---|
| Объявлено, ни одного использования, простой случай | Удалить |
| Объявлено, ни одного использования, экспортируется | Проверить публичный API → удалить если не API |
| Используется, не объявлено | Найти похожее имя → переименовать, или добавить объявление |
| Два похожих имени, одно мёртвое | Levenshtein/семантика → предложить слияние |
| Динамическое имя (`styles[name]`, template literal) | Флаг «динамика» → не трогать, отметить в отчёте |
| Side-effect импорт | Не трогать → отметить для ревью |
| Мёртвая ветка с `console.log` / `debugger` | Удалить точно |
| Мёртвая ветка с бизнес-логикой | Только предложить, не удалять автоматически |
| Тип/интерфейс без использований | Проверить re-export → удалить если нет |
| CSS-переменная без использований | Удалить если не в публичном design token файле |

## Часть 4. Инструкции для ИИ-агента

### 4.1 Порядок анализа файла

```
1. Определить тип файла: CSS / SCSS / TS / JS / JSX / TSX / HTML
2. Извлечь все сущности с их позицией (строка:колонка)
3. Для каждой сущности — классифицировать:
   a. Только объявлена
   b. Только используется
   c. И то и то
   d. Ни то ни то (в комментарии / строке)
4. Сформировать список кандидатов с типом проблемы
5. Для каждого кандидата — пройти проверки из Части 1-3
6. Сформировать отчёт с action: delete | rename | investigate | keep
```

### 4.2 Формат отчёта

Для каждой находки выдавать структуру:

```
НАХОДКА: <имя сущности>
ТИП: css-class | css-var | js-var | js-function | ts-type | js-import
ФАЙЛ: путь/к/файлу.css:42
ПРОБЛЕМА: unused | missing | renamed | duplicated | dead-branch
ЦЕПЬ: <откуда> → <куда> → <эффект>  (или "цепь не замыкается")
ДЕЙСТВИЕ: delete | rename <старое> → <новое> | investigate
ПРИЧИНА: <одно предложение почему>
УВЕРЕННОСТЬ: high | medium | low
```

Пример:

```
НАХОДКА: .card-wrapper
ТИП: css-class
ФАЙЛ: src/components/Card/Card.module.css:17
ПРОБЛЕМА: unused
ЦЕПЬ: Card.module.css → [нет входящих ссылок]
ДЕЙСТВИЕ: investigate → вероятно rename .card-wrapper → .cardContainer
ПРИЧИНА: В Card.tsx используется styles.cardContainer, в CSS объявлен только .card-wrapper — типичный артефакт переименования
УВЕРЕННОСТЬ: high
```

---

### 4.3 Приоритет проверок

Проверять в следующем порядке (от простого к сложному):

```
1. Мёртвые импорты (самый безопасный для удаления)
2. Локальные переменные без чтения
3. CSS классы без HTML/JS использований
4. CSS переменные без var(...)
5. Экспортируемые функции без внешних вызовов
6. Экспортируемые типы без импортов
7. Мёртвые ветки условий
8. Дублирующие функции / типы
```

---

### 4.4 Что НИКОГДА не удалять автоматически

```
- Всё что помечено @public, @api, @deprecated (запланированное удаление)
- Side-effect импорты (import 'x')
- Переменные с префиксом _ (intentionally unused)
- Классы которые могут прийти извне (SSR, CMS, markdown)
- Динамически собираемые имена (template literals, computed property names)
- Файлы *.d.ts целиком (декларации типов)
- Re-exports в index.ts / public-api.ts
- Всё в node_modules, dist, build
- Конфигурационные объекты (могут читаться рефлексией)
- Классы для e2e тестов (data-testid — не трогать даже если нет в unit тестах)
```

---

### 4.5 Grep-команды для ручной проверки

Когда уверенность `medium` или `low` — дать пользователю команды для проверки:

```bash
# Найти все использования класса
grep -rn "card-wrapper" src/ --include="*.{js,jsx,ts,tsx,html,vue,svelte}"

# Найти все импорты символа
grep -rn "import.*formatDate" src/

# Найти все var() для CSS-переменной
grep -rn "var(--color-brand)" src/

# Найти все вызовы функции
grep -rn "parseDate(" src/ --include="*.{js,ts,jsx,tsx}"

# Найти похожие имена (для артефактов переименования)
grep -rn "cardContainer\|card-container\|card_container\|cardWrapper\|card-wrapper" src/

# Проверить используется ли экспорт снаружи модуля
grep -rn "from.*utils" src/ | grep "OLD_API_URL"
```

## Часть 5. Чек-лист перед финальным удалением

Перед тем как удалить любую сущность — пройти:

- [ ] Поиск по всему проекту (`grep -rn`) дал ноль результатов
- [ ] Сущность не в публичном API (index.ts, package.json exports)
- [ ] Нет динамических ссылок (template literals, `require(variable)`)
- [ ] Нет в тестах (unit, e2e, storybook)
- [ ] Нет в документации (README, JSDoc, комментариях)
- [ ] Git blame показывает что это не «только что добавленное» (могут быть незафиксированные использования)
- [ ] Сборка проходит после удаления (`npm run build` / `tsc --noEmit`)
- [ ] Тесты зелёные после удаления

## Приложение: быстрые паттерны регулярок

```
CSS классы (объявленные):     /(?<![a-zA-Z0-9_-])\.([a-zA-Z_-][a-zA-Z0-9_-]*)\s*[{,]/g
```
