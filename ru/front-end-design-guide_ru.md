# Front-End Design Guide
## Полная система принципов дизайна и верстки

> Этот документ — система принципов, а не сборник решений.
> Каждый принцип — это проверочный вопрос, который можно задать к любому интерфейсу.
> Конкретные реализации меняются (фреймворки, дизайн-токены, компонентные библиотеки) —
> принципы остаются.

---

## Структура документа

```
I.   Восприятие          — как мозг обрабатывает визуальную информацию
II.  Иерархия            — порядок важности и внимания

XI.  Верификация         — как проверить любой UI
```

## I. Восприятие

Дизайн — это управление вниманием. Всё остальное вторично.

### I.1 Преаттентивные атрибуты

Мозг обрабатывает некоторые визуальные свойства до сознательного восприятия (~150–200 мс).
Это — самые мощные инструменты иерархии:

| Атрибут | Сила | Применение |
|---|---|---|
| Цвет (hue) | Высокая | Выделение статуса, категории |
| Размер | Высокая | Уровни заголовков, важность |
| Контраст (яркость) | Высокая | Первичный/вторичный контент |
| Движение | Очень высокая | Системные оповещения, прогресс |
| Форма | Средняя | Иконки, кнопки vs поля |
| Ориентация | Средняя | Стрелки, разделители |
| Позиция | Средняя | Навигация, подписи |

**Правило:** не смешивать больше 2–3 преаттентивных атрибутов для выделения одного элемента.
Когда всё кричит — не кричит ничто.

### I.2 Законы Гештальта

Мозг автоматически группирует визуальные элементы. Нарушение этих законов создаёт
когнитивный диссонанс — интерфейс воспринимается как «неудобный» без очевидной причины.

**Близость (Proximity)**
Элементы, расположенные рядом, воспринимаются как группа.
→ Между группами зазор должен быть больше, чем внутри группы.
→ Проверка: если убрать цвет и рамки — сохраняется ли группировка?

**Подобие (Similarity)**
Элементы с одинаковыми свойствами воспринимаются как связанные.
→ Одинаковая функция = одинаковый внешний вид. Разная функция = разный внешний вид.
→ Нарушение: кнопка и текстовая ссылка визуально одинаковы — путаница с функцией.

**Общая область (Common Region)**
Элементы внутри одного контейнера воспринимаются как группа.
→ Каждый дополнительный контейнер (рамка, фон) создаёт когнитивный сигнал «здесь граница».
→ Лишние контейнеры = лишние когнитивные усилия. Минимизировать.

**Непрерывность (Continuity)**
Взгляд следует по линии или кривой.
→ Горизонтальные ряды читаются слева направо. Вертикальные колонки — сверху вниз.
→ Нарушение порядка Tab = нарушение закона непрерывности в доступности.

**Замыкание (Closure)**
Мозг достраивает незавершённые фигуры.
→ Слегка обрезанный контент (peek pattern) сигнализирует о возможности скролла.
→ Можно показывать структуру меньшим количеством рамок.

**Фигура и фон (Figure-Ground)**
Элемент воспринимается либо как объект (фигура), либо как контекст (фон).
→ Модальные окна работают через подавление фона (overlay).
→ Карточки работают через отрыв от фона (shadow или contrast).

**Фокальная точка (Focal Point)**
Элемент, выбивающийся из паттерна, немедленно захватывает внимание.
→ Это принцип CTA-кнопки: одна точка контраста среди спокойного фона.

### I.3 Паттерны сканирования

**F-паттерн** (текстовые интерфейсы, списки):
Пользователь делает горизонтальное движение вверху страницы, затем вертикальное движение
вниз по левому краю, с короткими горизонтальными «прощупываниями».
→ Критическая информация — в первых словах каждого абзаца и первых строках страницы.

**Z-паттерн** (разреженные интерфейсы, лендинги):
Взгляд движется по диагонали Z — верхний левый → верхний правый → нижний левый → нижний правый.
→ Логотип слева, CTA справа, логические связи по диагонали.

**Паттерн Gutenberg** (сложные раскладки с несколькими блоками):
Первичная оптическая зона (верхний левый) и терминальная зона (нижний правый) — самые сильные.
→ Начало контента и финальный CTA — в этих зонах.

### I.4 Рабочая память

Рабочая память человека ограничена: 4 ± 1 смысловых единицы одновременно (Cowan, 2001).
Классическое «7 ± 2» (Miller) — это с применением «chunking»-а (группировки).

**Правила:**
- Не показывать больше 5–7 опций в одном меню без подгрупп.
- Связанные поля формы объединять в смысловые блоки с заголовком.
- Пагинация нужна не для перфоманса — для когнитивной разгрузки.
- Закон Хика (Hick's Law): время принятия решения растёт логарифмически с количеством опций.
  `T = a + b × log₂(n + 1)`
  → Прогрессивное раскрытие (progressive disclosure) — основной инструмент борьбы с Хиком.

## II. Иерархия

Иерархия — это ответ на вопрос: «что пользователь должен заметить первым?»

### II.1 Визуальный вес

Визуальный вес — относительная «тяжесть» элемента в восприятии. Управляется в порядке силы:

```
Размер > Контраст > Насыщенность цвета > Форма > Текстура > Позиция
```

**Принцип единственной доминанты:**
На каждом визуальном уровне (страница → раздел → карточка → строка) должен быть
один и только один самый тяжёлый элемент.
Два равновесных доминирующих элемента аннулируют иерархию.

**Проверка иерархии (3-second rule):**
Закрой глаза. Открой на 3 секунды. Закрой. Что ты запомнил?
Если запомнил 3+ разных вещи — иерархия не выстроена.

### II.2 Типы иерархии

**Информационная иерархия:** что важнее по смыслу.
**Визуальная иерархия:** что важнее по восприятию.
**Функциональная иерархия:** primary → secondary → tertiary actions.

Все три должны совпадать. Если самая важная информация визуально слабее второстепенной —
это нарушение.

### II.3 Иерархия действий

Любой интерфейс имеет не более трёх уровней действий:

| Уровень | Пример | Визуальное решение |
|---|---|---|
| Primary | «Оплатить», «Создать» | Filled button, высокий контраст |
| Secondary | «Редактировать», «Фильтр» | Outlined / ghost button |
| Tertiary | «Отмена», «Подробнее» | Text link или icon button |

**Правило:** на одном экране не должно быть двух primary-действий.
Если два действия одинаково важны — один из них не primary.

### II.4 Прогрессивное раскрытие

Показывать только то, что нужно прямо сейчас. Остальное — по запросу.

```
Уровень 0: то, что видно всегда    — основной контент
Уровень 1: открывается одним кликом — дополнительные опции, фильтры
Уровень 2: требует намерения        — расширенные настройки, детали
```

Нарушение: показывать всё сразу (toolbar с 12+ опциями, форма с 20+ полями).
Решение: accordion, dropdown, offcanvas, drawer, modal — структуры прогрессивного раскрытия.

## III. Типографика

Типографика — это не выбор шрифта. Это система отношений между текстовыми элементами.

### III.1 Модульная шкала

Размеры текста должны образовывать математическую прогрессию.
Стандартные коэффициенты:

| Имя | Коэффициент | Применение |
|---|---|---|
| Major Second | 1.125 | Плотные UI (таблицы, dashboards) |
| Minor Third | 1.2 | Большинство компонентов |
| Major Third | 1.25 | Документы, статьи |
| Perfect Fourth | 1.333 | Лендинги, маркетинг |
| Golden Ratio | 1.618 | Крупные заголовки, heroes |

**Правило:** в одном компоненте — не более 3–4 уровней шрифтовых размеров.
Больше уровней → иерархия разрушается.

### III.2 Межстрочный интервал (Leading)

```
Основной текст (body):   line-height 1.4–1.6
Заголовки:               line-height 1.1–1.25
Подписи, мета:           line-height 1.3–1.4
```

Слишком плотно: текст душит. Слишком разреженно: теряется связность абзаца.

### III.3 Мера (Measure)

Оптимальная длина строки — 45–75 символов для основного текста.
Короче 45: взгляд слишком часто переносится — утомляет.
Длиннее 75–85: взгляд теряет строку при переносе.

Инструмент: `max-width: 65ch` — физически ограничивает меру в CSS.

### III.4 Типографический ритм

Вертикальный ритм — постоянный шаг вертикального пространства на основе базовой единицы.
Если базовая единица = 4px:
```
Margin между параграфами: 16px (4 × 4)
Margin под заголовком:    24px (6 × 4)
Padding карточки:         16px или 24px
```
Хаотичные вертикальные отступы — самый частый признак отсутствия системы.

### III.5 Типографический контраст — не только размером

Контраст достигается комбинацией: размер / насыщенность (weight) / начертание / цвет / капитализация.
Не нужно увеличивать размер, чтобы выделить — достаточно поменять weight.

**Проверка:** убери размерные различия. Остаётся ли иерархия через weight и color? Если нет — система не полная.

## IV. Цвет

Цвет — последний инструмент иерархии, не первый.

### IV.1 WCAG-контраст

Любой текст на фоне должен пройти проверку на контраст:

| Тип контента | Минимум (AA) | Рекомендуется (AAA) |
|---|---|---|
| Основной текст (< 18pt) | 4.5 : 1 | 7 : 1 |
| Крупный текст (≥ 18pt или ≥ 14pt bold) | 3 : 1 | 4.5 : 1 |
| UI компоненты (границы, иконки) | 3 : 1 | — |
| Декоративные элементы | — | — |

Инструмент проверки: [contrast ratio = (L1 + 0.05) / (L2 + 0.05)], где L = относительная яркость.

### IV.2 Цвет как смысловой слой

Цвет должен нести функциональный смысл, а не просто эстетический:

```
Статусные значения (зарезервированы по умолчанию):
  Зелёный → успех, доступность, позитив
  Красный  → ошибка, опасность, удаление
  Жёлтый  → предупреждение, ожидание
  Синий    → информация, ссылка, интерактивность

Бренд-цвет → identity элементы (логотип, primary CTA)
```

**Правило цветовой независимости:** информация не должна передаваться только цветом.
Если убрать цвет — смысл должен сохраняться через форму, текст, иконку.
(Пример нарушения: красная/зелёная точка без подписи — непонятна дальтоникам)

### IV.3 Палитра

**Правило 60-30-10:**
- 60% — доминирующий нейтральный (фон, поверхности)
- 30% — вторичный (текст, рамки, разделители)
- 10% — акцентный (CTA, активные состояния, ссылки)

Больше 3–4 разных цветов в компоненте — признак отсутствия системы.

### IV.4 Токены цвета

Цвет не должен быть хардкодом. Система токенов:

```
Уровень 1 — Примитивы (raw palette):
  --color-blue-500: #3B82F6
  --color-red-500:  #EF4444

Уровень 2 — Семантические токены (what it means):
  --color-primary:     var(--color-blue-500)
  --color-error:       var(--color-red-500)
  --color-text:        var(--color-neutral-900)
  --color-surface:     var(--color-neutral-0)

Уровень 3 — Компонентные токены (where it's used):
  --btn-primary-bg:    var(--color-primary)
  --input-error-border:var(--color-error)
```

Смена темы = изменение уровня 2. Компонент не трогается.

## V. Пространство

### V.1 Базовая единица

Все отступы должны быть кратны одной базовой единице.
Стандарт: 4px или 8px.

```
Шкала на базе 4px:
  xs:  4px   — межсимвольные, тонкие разделители
  sm:  8px   — внутри элемента (padding icon button)
  md:  16px  — padding карточки, gap в группах
  lg:  24px  — между секциями компонента
  xl:  32px  — между крупными блоками
  2xl: 48px  — между разделами страницы
  3xl: 64px+ — hero sections, крупные отбивки
```

**Проверка:** каждый margin и padding делится на базовую единицу нацело? Если нет — есть «магические числа».

### V.2 Белое пространство как элемент

Белое пространство не «пустота» — это активный дизайн-элемент.
Функции белого пространства:
- Создаёт группы (proximity)
- Даёт «дышать» важному контенту
- Повышает читаемость текста
- Сигнализирует о качестве (luxury perception)

**Типичная ошибка:** «заполнить пространство» дополнительными элементами.
Решение: сначала спросить — зачем этот элемент, а не куда его поместить.

### V.3 Плотность контента

Интерфейсы различаются по плотности в зависимости от задачи:

| Тип | Плотность | Примеры |
|---|---|---|
| Потребление контента | Низкая | Статьи, product pages |
| Мониторинг данных | Высокая | Dashboards, таблицы |
| Ввод данных | Средняя | Формы |
| Навигация | Средняя | Меню, каталоги |

Плотность должна быть консистентной внутри типа интерфейса. Нельзя миксовать разреженный header и плотный main.

### V.4 Сетка

Сетка — это не ограничение, а система выравнивания.

**Колонки:** содержат контент. Ширина — процентная или фиксированная.
**Желоба (gutters):** отступы между колонками. Фиксированные (px), не процентные.
**Поля (margins):** отступы от края viewport. Адаптируются к breakpoints.

**Принцип выравнивания:** каждый элемент выровнен минимум по одной оси сетки.
Случайно выровненный элемент = ошибка.

## VI. Компонент

Компонент — самодостаточная единица интерфейса. Она решает одну задачу, не знает о контексте снаружи.

### VI.1 Атомарный дизайн (Atomic Design)

```
Атомы     — базовые HTML-элементы: кнопка, поле, иконка, лейбл
Молекулы  — группы атомов: поисковая строка (input + button + icon)
Организмы — группы молекул: header, форма, карточка отзыва
Шаблоны  — расположение организмов: макет страницы
Страницы  — шаблоны с реальным контентом
```

**Правило:** компонент не должен «знать» о своём расположении на странице.
Нарушение: `margin-top: 40px` внутри компонента — это знание о внешнем контексте.

### VI.2 Изоляция компонента

**CSS-изоляция:** все стили компонента скоупированы на его корневой элемент.
```css
/* Правильно: */
.card { ... }
.card__title { ... }

/* Неправильно: */
h2 { color: blue; } /* глобальный сайд-эффект */
```

**JS-изоляция (Law Zero):** компонент не использует глобальные ID-хуки.
```js
// Неправильно:
document.getElementById('submit-btn')

// Правильно:
this.root.querySelector('[data-action="submit"]')
```

**Namespace изоляция:** все кастомные data-атрибуты, CSS-переменные, классы — с префиксом компонента.
```
Плохо:  data-filter="5"        --color-primary
Хорошо: data-reviews-filter="5" --reviews-star-color
```

### VI.3 Машина состояний компонента

Каждый интерактивный компонент — это конечный автомат. Состояния должны быть явными и полными:

```
Обязательные состояния для большинства компонентов:
  idle       — покой, начальное состояние
  hover      — курсор над элементом
  focus      — клавиатурный фокус
  active     — нажатие/взаимодействие
  disabled   — недоступен
  loading    — ожидание ответа
  error      — ошибка
  success    — успех / завершено
  empty      — нет данных для отображения
```

**Правило:** если состояние визуально не отличается от другого — оно не реализовано, а не отсутствует.
Нарушение: disabled-кнопка выглядит как active-кнопка.

### VI.4 Адаптивность

Компонент должен корректно работать в диапазоне контейнеров, не только на стандартных breakpoints.
Mobile-first: начинать с минимального контейнера, расширять через `min-width`.

```css
/* Неправильно — desktop-first: */
.grid { display: grid; grid-template-columns: 1fr 1fr 1fr; }
@media (max-width: 768px) { .grid { grid-template-columns: 1fr; } }

/* Правильно — mobile-first: */
.grid { display: grid; grid-template-columns: 1fr; }
@media (min-width: 768px) { .grid { grid-template-columns: 1fr 1fr 1fr; } }
```

## VII. Взаимодействие

### VII.1 Законы взаимодействия

**Закон Фиттса (Fitts's Law):**
Время достижения цели = f(расстояние / размер цели).
→ Минимальные размеры интерактивных элементов:
  - Apple HIG: 44 × 44 pt
  - Google Material: 48 × 48 dp
  - WCAG 2.5.5 (AAA): 44 × 44 CSS px
→ Элементы, которые используются часто — должны быть большими и близко.
→ Деструктивные действия (удалить) — должны быть маленькими и далеко.

**Закон Хика (Hick's Law):**
Уже описан в разделе I.4. Применение к взаимодействию:
→ Toolbar с 10+ опциями — нарушение. Решение: группировка + прогрессивное раскрытие.
→ Форма с 15+ полями — нарушение. Решение: шаги (wizard), логические секции.

**Эффект позиционирования (Serial Position Effect):**
Лучше всего запоминаются первый и последний элементы списка (primacy + recency).
→ CTA-кнопка в конце формы — правильно (recency).
→ Самый важный пункт меню — первый или последний.

### VII.2 Обратная связь (Feedback)

Nielsen's Heuristic #1: видимость системного статуса.
Каждое действие пользователя должно иметь немедленную видимую реакцию:

```
< 100ms    — «мгновенно», реакция не нужна визуально
100–300ms  — нужен visual feedback (hover state, ripple)
300–1000ms — нужен индикатор загрузки (spinner, skeleton)
> 1s       — нужен прогресс-бар + возможность отмены
> 10s      — нужно уведомление + возможность уйти и вернуться
```

**Типы feedback:**
- Цвет/иконка — немедленный (hover, pressed)
- Microanimation — подтверждение действия (лайк, добавление в корзину)
- Toast / snackbar — асинхронный результат (сохранено, ошибка сети)
- Skeleton screen — загрузка контента
- Progress bar — долгие операции

### VII.3 Анимация

Анимация должна быть мотивированной. Каждая анимация отвечает на вопрос: «что она объясняет?»

**Функции анимации:**
1. Ориентация — показывает, откуда появился / куда ушёл элемент (modal, drawer)
2. Связь — показывает отношение между состояниями (expand, accordion)
3. Подтверждение — подтверждает завершение действия (чекмарк, shake на ошибку)
4. Прелоадинг — делает ожидание терпимым (skeleton, spinner)
5. Деlight — добавляет характер (bounce, spring — только там, где уместно)

**Параметры анимации:**
```
Микроинтеракции (hover, toggle):  100–200ms,  ease-in-out
Появление элемента:               200–300ms,  ease-out
Уход элемента:                    150–200ms,  ease-in
Сложные переходы (page):          300–500ms,  custom spring
```

Длиннее 500ms — анимация перестаёт казаться отзывчивой.
Анимация для анимации (без функции) — паразитная.

**prefers-reduced-motion:** все анимации должны быть отключаемы:
```css
@media (prefers-reduced-motion: reduce) {
  * { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; }
}
```

### VII.4 Microinteraction Anatomy (Dan Saffer)

```
Trigger   — что запускает взаимодействие (клик, hover, загрузка данных)
Rules     — что происходит (логика)
Feedback  — как это отображается (визуал, звук, тактильность)
Loops     — повторяется ли, когда заканчивается
```

Хорошая микроинтеракция прозрачна — пользователь не думает о ней, просто «всё работает правильно».

## VIII. Доступность

Доступность — не фича. Это базовое требование качества, как отсутствие ошибок компиляции.

### VIII.1 Четыре принципа WCAG (POUR)

**Perceivable** (воспринимаемый):
- Alt-текст для изображений
- Субтитры для видео
- Достаточный цветовой контраст
- Текст не зашит в изображение

**Operable** (управляемый):
- Вся функциональность доступна с клавиатуры
- Нет ловушек фокуса (кроме модалов — там это правильно)
- Достаточное время для взаимодействия
- Нет вспышек > 3 раз/сек

**Understandable** (понятный):
- Язык страницы указан (`lang="ru"`)
- Поведение компонентов предсказуемо
- Ошибки формы описывают проблему и решение
- Лейблы у всех полей

**Robust** (надёжный):
- Валидный HTML
- Совместимость с AT (assistive technology)
- Правильные ARIA-роли

### VIII.2 Семантический HTML — фундамент

Правило: если для элемента есть нативный HTML-тег — использовать его, не `div`.

```
Интерактивность:  <button>, <a>, <input>, <select>, <textarea>
Структура:        <header>, <main>, <nav>, <aside>, <footer>, <section>, <article>
Типографика:      <h1>–<h6>, <p>, <ul>, <ol>, <li>, <blockquote>
Данные:           <table>, <caption>, <thead>, <tbody>, <th scope>
Медиа:            <figure>, <figcaption>, <picture>
```

Нативные элементы дают бесплатно: keyboard interaction, focus management, ARIA roles, browser defaults.
`<div>` + JS-обработчик — всё это нужно добавлять вручную.

### VIII.3 ARIA — дополнение, не замена

ARIA расширяет семантику там, где нативного HTML не хватает. Первое правило ARIA:
> Если можно использовать нативный HTML-элемент с нужной семантикой — используй его.

```
Роли:        role="dialog", role="list", role="alert", role="tabpanel"
Состояния:   aria-expanded, aria-checked, aria-pressed, aria-selected
Свойства:    aria-label, aria-labelledby, aria-describedby, aria-controls
Live-регионы: aria-live="polite", aria-live="assertive"
```

**aria-label vs aria-labelledby:**
- `aria-labelledby` ссылается на существующий видимый текст — предпочтительно.
- `aria-label` для элементов без видимого лейбла (иконка-кнопка, поиск без визуального заголовка).

### VIII.4 Фокус-менеджмент

```
Фокус-индикатор:  всегда видим и контрастен (:focus-visible)
Порядок фокуса:   соответствует логическому порядку DOM (tabindex="0" только при необходимости)
Ловушка фокуса:   модалы и диалоги — фокус не выходит пока открыто
Возврат фокуса:   при закрытии модала — фокус возвращается к триггеру
Skip-links:       <a href="#main">Перейти к содержимому</a> — первый элемент страницы
```

`tabindex > 0` — почти всегда ошибка. Переосмысли структуру DOM.

## IX. CSS-архитектура

CSS — это не таблица стилей. Это язык описания дизайн-системы.

### IX.1 Каскад и специфичность

Специфичность (от низкой к высокой):
```
Селекторы тегов:          0-0-1   (h1, p, div)
Классы, атрибуты:         0-1-0   (.card, [type="text"])
ID:                       1-0-0   (#header)
Inline styles:            1-0-0-0
!important:               вне системы
```

**Принцип минимальной специфичности:**
Писать стили с наименьшей необходимой специфичностью.
- Не использовать ID-селекторы для стилей.
- Не использовать `!important` кроме utility-классов (последний слой).
- Не вкладывать селекторы глубже 2–3 уровней.

### IX.2 Система токенов

Трёхуровневая архитектура (см. раздел IV.4):

```
Глобальные токены (primitives):
  Все возможные значения дизайн-системы.
  Никогда не используются напрямую в компонентах.

Семантические токены (decisions):
  Значение привязано к контексту применения.
  Это то, что меняет тема.

Компонентные токены:
  Конкретный компонент ссылается на семантические токены.
  Переопределяются только для локальных вариантов.
```

**Правило:** токен всегда ссылается на токен уровнем выше, не на примитив напрямую.
```css
/* Плохо: */
.button-primary { background: #3B82F6; }

/* Хорошо: */
.button-primary { background: var(--button-primary-bg); }
--button-primary-bg: var(--color-primary);
--color-primary: var(--color-blue-500);
--color-blue-500: #3B82F6;
```

### IX.3 Стратегии скоупинга

**BEM (Block Element Modifier):**
```css
.card { }                  /* Block */
.card__title { }           /* Element */
.card--featured { }        /* Modifier */
.card__title--truncated { } /* Element + Modifier */
```

**Utility-first (Tailwind-подход):**
Атомарные классы применяются прямо в HTML. Нет кастомного CSS без необходимости.
Сильная сторона: нет проблем со специфичностью, легко читать.
Слабая сторона: HTML захламлён, трудно экстрагировать паттерны.

**Scoped / CSS Modules:**
Каждый компонент получает уникальные хеш-суффиксы к классам. Нет коллизий.
Сильная сторона: полная изоляция. Слабая: сложнее override из темы.

**Component-scoped (наш подход):**
Все дочерние классы — с префиксом компонента.
```css
.reviews-container { }         /* Корень, CSS-скоуп */
.reviews-container .reviews-card { }   /* Дочерние — всегда через корень */
```

### IX.4 Принципы CSS

**DRY через кастомные свойства, не через дублирование:**
Повторяющееся значение → токен. Повторяющийся паттерн → класс.

**Не бороться с браузером:**
Flexbox и Grid решают 95% задач раскладки.
`position: absolute` — только когда смысл действительно абсолютный.
Float в 2025 году — только для text-wrap.

**Логические свойства вместо физических:**
```css
/* Физические — сломаются в RTL: */
padding-left: 16px; margin-right: 8px;

/* Логические — работают в любом направлении: */
padding-inline-start: 16px; margin-inline-end: 8px;
```

## X. Инженерия компонента

### X.1 Разделение ответственности

```
HTML  — структура и семантика
CSS   — внешний вид и анимация
JS    — поведение и состояние
```

JS не должен писать inline styles (кроме CSS-custom properties).
CSS не должен содержать логику (кроме `:has()`, `:is()`, `:where()` структурных селекторов).
HTML не должен содержать presentation-атрибуты (`style="..."`, устаревшие `align`, `bgcolor`).

### X.2 Хуки в HTML

Разделение по типу атрибута:

```
class="..."          — стилевые хуки (CSS-цель)
id="..."             — ARIA-якоря (for=, aria-labelledby=), URL-якоря, единственный экземпляр
data-ref="..."       — JS-ссылки на DOM-узлы (кэшируются при инициализации)
data-action="..."    — JS-триггеры событий (делегированный listener на корне)
data-state="..."     — состояние, видимое и для JS, и для CSS
data-*-custom="..."  — данные с префиксом компонента
```

Запрет: JS-классы (`js-*`) как хуки событий — устаревший паттерн.
Запрет: ID как CSS-селекторы — ломает специфичность.
Запрет: смешивать роли (`id` используется и для CSS, и для ARIA, и для JS) — коллизии.

### X.3 Делегирование событий

```js
// Плохо — listener на каждом элементе:
document.querySelectorAll('.btn').forEach(btn =>
  btn.addEventListener('click', handler)
);

// Хорошо — один listener на корне:
root.addEventListener('click', e => {
  const btn = e.target.closest('[data-action]');
  if (!btn) return;
  switch (btn.dataset.action) { ... }
});
```

Преимущества: работает с динамически добавленными элементами, одна точка регистрации,
легко очистить через AbortController.

### X.4 AbortController для cleanup

```js
_bindEvents() {
  const sig = { signal: this._abort.signal };
  this._root.addEventListener('click', this._onClick.bind(this), sig);
  document.addEventListener('keydown', this._onKey.bind(this), sig);
}

destroy() {
  this._abort.abort(); // одна строка чистит все listeners
}
```

### X.5 Безопасность

**XSS (Cross-Site Scripting):**
```js
// Опасно — интерполяция в innerHTML:
el.innerHTML = `<p>${userInput}</p>`;

// Безопасно — escape перед вставкой:
_esc(s) { return String(s).replace(/[&<>"']/g, c => ({'&':'&amp;','<':'&lt;','>':'&gt;','"':'&quot;',"'":'&#39;'}[c])); }
el.innerHTML = `<p>${this._esc(userInput)}</p>`;

// Или использовать textContent для чистого текста:
el.textContent = userInput;
```

**URL-валидация:**
```js
_isSafeUrl(url) {
  return typeof url === 'string' && /^https?:\/\//i.test(url);
}
```

**Валидация входных данных:**
Любые данные из внешних источников (API, localStorage, URL-параметры) — валидировать
до использования. Ключи объектов проверять против whitelist.

### X.6 Производительность рендера

**Layout thrashing:**
```js
// Плохо — чередование read/write вызывает forced reflow:
el.style.height = el.offsetHeight + 'px';
el2.style.height = el2.offsetHeight + 'px';

// Хорошо — сначала все reads, потом все writes:
const h1 = el.offsetHeight;
const h2 = el2.offsetHeight;
el.style.height = h1 + 'px';
el2.style.height = h2 + 'px';
```

**Batch DOM updates:**
```js
// Плохо — множественные вставки в DOM:
items.forEach(item => list.appendChild(createEl(item)));

// Хорошо — один DocumentFragment:
const frag = document.createDocumentFragment();
items.forEach(item => frag.appendChild(createEl(item)));
list.replaceChildren(frag);
```

**CSS transitions вместо JS-анимаций:**
Браузер оптимизирует `transform` и `opacity` через compositor — без reflow/repaint.
Анимировать: `transform`, `opacity`.
Не анимировать: `width`, `height`, `top`, `left`, `margin` — вызывают layout.

## XI. Верификация

Набор вопросов для аудита любого интерфейса.

### XI.1 Перцептивный аудит

```
□ Правило 3 секунд: что запоминается первым после 3 секунд просмотра?
  → Это должен быть главный контент / primary action.

□ Иерархия: у каждого раздела есть один доминирующий элемент?
  → Нет двух одинаково «тяжёлых» элементов рядом?

□ Группировка: убрать рамки и цвет — сохраняется ли структура через пространство?
  → Близость создаёт группы без decoration?

□ Сканируемость: критическая информация в первых словах строк?
  → F-паттерн соблюдён для текстового контента?

□ Лишние контейнеры: каждый card/box/border обоснован?
  → Нет рамок «для красоты» без группирующей функции?
```

### XI.2 Типографический аудит

```
□ Шкала: количество уровней размера в компоненте ≤ 4?
□ Ритм: вертикальные отступы кратны базовой единице?
□ Мера: ширина текстового блока ≤ 75 символов?
□ Иерархия без размера: убрать size-различия — сохраняется иерархия через weight/color?
□ Контраст: все тексты прошли WCAG AA (минимум 4.5:1)?
```

### XI.3 Цветовой аудит

```
□ Контраст: весь текст и UI-элементы прошли WCAG-проверку?
□ Независимость: смысл сохраняется без цвета?
□ Семантика: цвет используется консистентно (зелёный всегда = успех)?
□ Токены: все цвета через переменные, ноль хардкода?
□ Палитра: не более 3–4 цветов в компоненте?
```

### XI.4 Пространственный аудит

```
□ Единица: все margin/padding кратны базовой единице?
□ Магических чисел нет: нет значений типа 13px, 37px, 5px?
□ Mobile: touch targets ≥ 44px × 44px?
□ Плотность: консистентна внутри компонента?
```

### XI.5 Компонентный аудит

```
□ Состояния: реализованы idle, hover, focus, disabled, loading, error, empty?
□ Изоляция: компонент не задаёт внешние отступы (margin вовне)?
□ Префикс: все data-атрибуты и CSS-переменные с namespace?
□ Скоуп: все CSS-правила привязаны к корневому элементу компонента?
□ Нет глобальных ID-хуков в JS?
□ Нет inline styles (кроме CSS custom properties)?
```

### XI.6 Доступность — аудит

```
□ Семантика: все интерактивные элементы — нативные button/a/input?
□ ARIA: все необходимые aria-label/labelledby/describedby есть?
□ Контраст: прошли WCAG AA?
□ Клавиатура: вся функциональность доступна через Tab/Enter/Escape/Arrows?
□ Фокус: :focus-visible стиль видим и контрастен?
□ Live-регионы: динамический контент объявляется screen reader?
□ Порядок фокуса: логический, соответствует визуальному?
□ Размеры: touch targets ≥ 44px?
```

### XI.7 CSS-архитектура — аудит

```
□ Специфичность: нет ID-селекторов в CSS?
□ !important: только в utility-классах последнего слоя?
□ Токены: все значения через CSS custom properties?
□ Темизируемость: смена переменных меняет весь внешний вид без правки правил?
□ Скоуп: каждый дочерний селектор идёт через корневой класс?
□ Глобальные утилиты: не вшиты в компонентный CSS?
```

### XI.8 Инженерный аудит

```
□ Разделение: CSS-только анимации для transform/opacity?
□ Делегирование: один listener на корне, не на каждом элементе?
□ XSS: весь пользовательский контент escaped перед innerHTML?
□ URL-валидация: все внешние URL проверены перед использованием в src/href?
□ Cleanup: все listeners удаляются при destroy компонента?
□ DOM-батчинг: множественные обновления через DocumentFragment?
□ Нет layout thrashing: reads и writes не перемешаны?
```

## Приложение: Чеклист одной страницей

```
ВОСПРИЯТИЕ
  □ 3-second rule: главное — первое
  □ Одна доминанта на раздел
  □ Группировка через пространство, не только рамки

ИЕРАРХИЯ
  □ Один primary action на экран
  □ Progressive disclosure для сложных интерфейсов
  □ Информационная = Визуальная = Функциональная иерархия

ТИПОГРАФИКА
  □ ≤ 4 размерных уровня в компоненте
  □ Вертикальный ритм на базовой единице
  □ Мера ≤ 75 символов

ЦВЕТ
  □ Все тексты: WCAG AA (4.5:1)
  □ UI-элементы: WCAG AA (3:1)
  □ Смысл без цвета сохраняется
  □ Все цвета через токены

ПРОСТРАНСТВО
  □ Все отступы кратны базовой единице
  □ Touch targets ≥ 44px

КОМПОНЕНТ
  □ Все состояния реализованы
  □ Нет внешних margin
  □ Namespace префикс на всём

ВЗАИМОДЕЙСТВИЕ
  □ Каждое действие имеет немедленный feedback
  □ Анимация мотивирована функцией
  □ prefers-reduced-motion учтён

ДОСТУПНОСТЬ
  □ Нативный HTML для всего интерактивного
  □ Все поля с лейблами
  □ Полная клавиатурная навигация
  □ :focus-visible стиль есть

CSS
  □ Нет ID в CSS
  □ Нет !important (кроме utils)
  □ Все стили скоупированы на корень
  □ Токены на трёх уровнях

ИНЖЕНЕРИЯ
  □ Escape всего пользовательского ввода
  □ Один делегированный listener
  □ Cleanup при уничтожении
```
