I

Восприятие

Дизайн — это управление вниманием. Мозг обрабатывает визуальные атрибуты ещё до сознательного восприятия (150–200 мс). Нарушение этих законов делает интерфейс «неудобным» без очевидной причины.

Правило Преаттентивные атрибуты — не более 2–3 для одного элемента
Плохо

Жёлтый фон + красный текст + огромный размер + жирность + курсив + caps + синяя рамка + анимация — 7 атрибутов одновременно. Когда всё кричит — ничто не слышно.

Хорошо

Один выделенный элемент (цвет + размер). Второй — тихий. Иерархия очевидна за 150 мс. Не нужно думать, что важнее.

Гештальт Близость (Proximity) — пространство создаёт группы
Плохо
Имя
Войти
Email
Регистрация
Пароль
Забыли пароль?

Все элементы на одном расстоянии. Мозг не понимает группировку — какие поля относятся к логину, а какие к регистрации?

Хорошо
Логин
Email
Пароль
Забыли пароль?
Новый пользователь
Имя
Регистрация

Расстояние между группами больше, чем внутри. Граница читается без рамок и цвета — только через пространство.

Сканирование F-паттерн — важное в начале строки и сверху
Плохо

Наш сервис помогает командам автоматизировать процессы уже 15 лет.

Работаем с клиентами по всему миру. Первый месяц бесплатно, затем скидка 30% при оплате годовой подписки.

Ключевые данные («15 лет», «30%») в конце предложений. При быстром сканировании их не заметят.

Хорошо

15 лет разработки ПО для международных клиентов.

Сэкономьте 30% при оплате годовой подписки сегодня.

Самое важное — в начале строки. Пользователь сканирует взглядом вниз по левому краю и сразу видит ценность.

Закон Хика Прогрессивное раскрытие против паралича выбора T = a + b × log₂(n+1)
Плохо

10 равнозначных опций одновременно. Время принятия решения растёт логарифмически — пользователь замирает.

Хорошо

Главное действие видно сразу. Остальное скрыто до явного намерения. Нажмите «Ещё опции» чтобы увидеть раскрытие.

II

Иерархия

Иерархия отвечает на один вопрос: «что пользователь должен заметить первым?» На каждом визуальном уровне должен быть один и только один самый весомый элемент.

Правило На экране не должно быть двух primary-действий
Плохо
Два одинаково весомых CTA

Два заполненных кнопки одного размера и цвета уничтожают иерархию. Пользователь теряется — глаз не знает, куда идти.

Хорошо
Primary → Secondary → Tertiary

Один заполненный (primary), один контурный (secondary), один текстовый (tertiary). Иерархия действий читается мгновенно.

Тест Правило 3 секунд — что запоминается при первом взгляде?
Плохо — 3 одинаково «тяжёлых» блока
СКИДКА 30%
НОВИНКА!
ТОП ПРОДАЖ

Три ярких блока с одинаковым весом. Закройте глаза на 3 секунды и откройте — что вы запомнили? Всё кричит одновременно.

Хорошо — один доминирующий элемент
Скидка 30%
Новинка
Топ продаж

Один крупный доминирующий элемент, остальные подчинённые. За 3 секунды запоминается именно он — и это именно то, что нужно.

III

Типографика

Типографика — не выбор шрифта, а система отношений между текстовыми элементами. Размеры, межстрочный интервал, длина строки и ритм — всё это управляется через математику, не интуицию.

Масштаб Размеры текста — математическая прогрессия, не случайные числа
Плохо — хаотичные размеры

Заголовок страницы

Подзаголовок раздела

Заголовок карточки

Основной текст описания.

/* Хаотичные px без системы */ h2 { font-size: 22px; } h3 { font-size: 21px; } h4 { font-size: 19px; } p { font-size: 15px; }

Разница между h2 и h3 — всего 1px. Иерархия не читается. Нет ощущения системы.

Хорошо — модульный масштаб 1.25 (Major Third)

Заголовок страницы

Подзаголовок раздела

Основной текст описания компонента.

/* Прогрессия: 14 × 1.25 × 1.25 = 21.8 ≈ 22px */ /* Базовый: 14px, коэф. Major Third: 1.25 */ --text-base: 14px; /* p */ --text-lg: 17.5px; /* sub */ --text-xl: 21.9px; /* h2 */

Каждый уровень = предыдущий × 1.25. Иерархия читается без цветов и весов. Максимум 3–4 уровня в одном компоненте.

Мера Длина строки: 45–75 символов для основного текста
Плохо — слишком широко
Пользователь сканирует текст по F-паттерну. Когда строка слишком длинная, глаз теряется при возврате на следующую строку и начинает читать ту же строку повторно.

Ширина не ограничена. Глаз устаёт при переходе на следующую строку — это читабельность ниже нуля.

Хорошо — max-width: 65ch
Пользователь сканирует текст по F-паттерну. Когда строка слишком длинная, глаз теряется при возврате на следующую строку и начинает читать ту же строку повторно.
p { max-width: 65ch; }

65ch = ширина 65 символов текущего шрифта. Физически ограничивает строку без зависимости от px.

Ритм Вертикальный ритм — все отступы кратны базовому юниту
Плохо — магические числа
Заголовок раздела
Описание
Ещё элемент
Последний
margin-bottom: 3px; /* магия */ margin-bottom: 11px; /* откуда? */ margin-bottom: 5px; /* почему? */

3px, 11px, 5px — числа без системы. Хаотичный ритм создаёт ощущение небрежности даже при хорошем контенте.

Хорошо — шаг 4px
Заголовок раздела
Описание
Ещё элемент
Последний
/* Шаг 4px: 4, 8, 12, 16, 24, 32... */ --space-sm: 8px; --space-md: 16px; --space-lg: 24px; margin-bottom: var(--space-md);

Все отступы кратны 4px. Равномерный ритм воспринимается как аккуратность и профессионализм — даже если пользователь не понимает, почему.

IV

Цвет

Цвет — последний инструмент иерархии, не первый. Он несёт функциональный смысл, а не только эстетический. Без выполнения требований контрастности WCAG — это не дизайн, это нарушение доступности.

WCAG Контрастность текста — минимум 4.5:1 для обычного текста WCAG AA
Плохо — недостаточный контраст
Этот текст плохо читается на фоне
Контраст: ~1.4:1 ✗ (минимум 4.5:1)

Серый текст на светло-сером фоне. Нечитаемо для 8% мужчин с цветовой слепотой и для всех при ярком освещении или плохом экране.

Хорошо — высокий контраст
Этот текст отлично читается на тёмном фоне
Контраст: ~21:1 ✓ (максимально возможный)

Белый текст на тёмном фоне. Проходит WCAG AAA. Используйте браузерные инструменты (DevTools → Accessibility) для проверки.

Доступность Смысл не должен передаваться только цветом
Плохо — только цвет
Сервер 1
Сервер 2
Сервер 3

Зелёная/красная точка без текстового пояснения. Для человека с дейтеранопией (красно-зелёная слепота) все точки выглядят одинаково.

Хорошо — цвет + иконка + текст
✓ Сервер 1 — в сети
✕ Сервер 2 — ошибка
✓ Сервер 3 — в сети

Цвет + иконка + текст. Уберите цвет — смысл сохраняется. Это правило независимости от цвета.

Архитектура Трёхуровневая система токенов цвета
Плохо — захардкоженные значения
.button { background: #3B82F6; } .link { color: #3B82F6; } .badge { border-color: #3B82F6; } /* Сменить тему = найти 47 вхождений #3B82F6 */

Один и тот же hex в десятках мест. Смена темы = рефакторинг всего кода. Нет возможности отличить «синий для брендинга» от «синий для информации».

Хорошо — токены в 3 уровня
Примитив --color-blue-500: #3B82F6
↓ ссылается
Семантика --color-primary: var(--color-blue-500)
↓ ссылается
Компонент --btn-bg: var(--color-primary)

Смена темы = изменить только уровень 2. Компоненты не трогаются. Каждый токен ссылается на уровень выше, никогда на примитив напрямую.

V

Пространство

Пространство — не пустота. Это активный элемент дизайна. Все отступы должны быть кратны единому базовому юниту (4px или 8px). Случайные значения — признак отсутствия системы.

Базовый юнит Mobile-first адаптивность — min-width, не max-width
Плохо — Desktop First (max-width)
Карточка 1
Карточка 2
Карточка 3
.grid { grid-template-columns: 1fr 1fr 1fr; } /* Потом «ломаем» под мобайл: */ @media (max-width: 768px) { .grid { grid-template-columns: 1fr; } }

Начинаем с десктопа и «урезаем» под мобайл. Мышление от большего к меньшему — потом не хватает контента и появляются костыли.

Хорошо — Mobile First (min-width)
Карточка 1
Карточка 2
/* Начинаем с мобайла: */ .grid { grid-template-columns: 1fr; } /* Расширяем на больших экранах: */ @media (min-width: 768px) { .grid { grid-template-columns: 1fr 1fr 1fr; } }

Начинаем с наименьшего контейнера, расширяем через min-width. Компонент корректно работает в любом размере без переопределений.

VI

Компоненты

Компонент — самодостаточная единица интерфейса. Он решает одну задачу и не знает о внешнем контексте. Каждое интерактивное состояние должно быть реализовано явно.

Состояния Машина состояний — каждое состояние визуально различимо
Плохо — disabled выглядит как active

Два состояния неразличимы. Пользователь не понимает, почему кнопка не реагирует — активна она или нет.

Хорошо — 8 различимых состояний
idle
hover
focus
active
disabled
loading
error
success

Если состояние визуально неотличимо от другого — оно не реализовано, оно просто отсутствует.

Изоляция Компонент не знает о своём внешнем контексте
Плохо — внешние отступы внутри компонента
/* Компонент знает, где он живёт */ .card { margin-top: 40px; /* ← НАРУШЕНИЕ */ margin-left: 24px; /* ← НАРУШЕНИЕ */ padding: 16px; }

margin-top/left — это знание о внешнем контексте. Перенесите компонент в другое место — вёрстка сломается.

Хорошо — только внутренние отступы
/* Компонент изолирован */ .card { padding: 16px; /* ← только внутри */ } /* Контекст управляет расположением */ .page-grid { gap: 24px; /* ← снаружи */ }

Компонент управляет только своим внутренним пространством. Внешние отступы — ответственность родителя (gap, margin от контейнера).

Именование Изоляция через пространство имён — префикс компонента
Плохо — глобальные имена
data-filter="5" --color-primary .title .card /* Коллизия: у двух компонентов есть .title */ /* Чьи стили применятся? */
Хорошо — префикс компонента
data-reviews-filter="5" --reviews-star-color .reviews-title .reviews-card /* Нет коллизий, сразу понятно, кому принадлежит каждый элемент */
VII

Взаимодействие

Каждое действие пользователя должно иметь немедленный видимый отклик. Анимация должна быть мотивированной — объяснять что-то, а не украшать.

Обратная связь Время отклика определяет тип визуальной обратной связи
< 100 мс «Мгновенно» — доп. обратная связь не нужна (hover-смена цвета достаточна)
100–300 мс Нужен hover-state, ripple-эффект, смена курсора
300 мс–1 с Нужен индикатор загрузки: spinner или skeleton screen
> 1 секунды Прогресс-бар + возможность отменить операцию
> 10 секунд Уведомление + возможность уйти и вернуться
Анимация Скорость анимации — 100–300 мс. Медленнее — перестаёт быть отзывчивым
Плохо — слишком медленно (2 сек)

transition: 2s ease

Анимация длиннее 500 мс перестаёт ощущаться как отклик — становится препятствием. Пользователь ждёт, пока элемент «соизволит» отреагировать.

Хорошо — 150 мс (micro-interaction)

transition: 0.15s ease-out

150 мс — граница «мгновенного» отклика. Анимация подтверждает действие, не задерживает. Параметры: micro-interaction 100–200 мс, ease-in-out.

Перфоманс Анимируйте только transform и opacity — без reflow
Плохо — анимация layout-свойств
/* Вызывают Layout + Paint + Composite */ .element { transition: width 0.3s, height 0.3s, margin 0.3s, top 0.3s; }

Анимация width, height, top, margin вызывает полный пересчёт layout на каждый кадр. Дёргается на слабых устройствах.

Хорошо — только compositor-слой
/* Compositor Layer — нет reflow/repaint */ .element { transition: transform 0.2s ease-out, opacity 0.2s ease-out; } /* transform заменяет position: */ .element--visible { transform: translateX(0); opacity: 1; }

transform и opacity обрабатываются GPU в отдельном compositor-слое. Никакого reflow, 60 fps на любом устройстве.

VIII

Доступность

Доступность — не фича, а базовое требование к качеству. Как отсутствие ошибок компиляции. Нативный HTML даёт клавиатурную навигацию, ARIA-роли и управление фокусом бесплатно.

Семантика Нативные HTML-элементы вместо div с обработчиками
Плохо — div вместо button
Отправить форму

Tab — не фокусируется. Enter — не срабатывает. Screen reader не говорит «кнопка».

/* div требует всё добавлять вручную */ <div onclick="submit()" tabindex="0" role="button" onkeydown="..."> Отправить </div>

<div> не фокусируется по Tab, не реагирует на Enter/Space, не объявляется screen reader как кнопка. Всё это нужно добавлять вручную через ARIA и JS.

Хорошо — нативный button

Нажмите Tab → Enter для активации. Попробуйте с клавиатуры.

/* Всё из коробки бесплатно: */ <button type="submit"> Отправить </button> /* ✓ Tab-фокус ✓ Enter/Space ✓ ARIA role=button ✓ disabled-состояние */

Нативный <button> даёт всё бесплатно. Попробуйте Tab → Enter — кнопка выше работает с клавиатуры.

Фокус :focus-visible всегда должен быть видим и высококонтрастным
Плохо — outline: none
button:focus { outline: none; /* ← НАРУШЕНИЕ */ }

Уберите outline для «красоты» — и клавиатурный пользователь не видит, где он находится. WCAG нарушение, юридический риск.

Хорошо — высококонтрастный :focus-visible
/* :focus-visible — только при навигации с клавиатуры, не при клике мышью */ button:focus-visible { outline: 3px solid #fff; outline-offset: 3px; }

:focus-visible показывается только при клавиатурной навигации, не при клике мышью. Нажмите Tab и наведите на кнопку выше — кольцо фокуса должно быть заметным.

Форма Все поля формы обязаны иметь явные <label>
Плохо — placeholder вместо label

Placeholder исчезает при вводе. Screen reader не объявит назначение поля. Placeholder — подсказка, не метка.

Хорошо — явный <label>
<label for="email">Email адрес</label> <input type="email" id="email" placeholder="user@example.com">

Label всегда виден, кликабелен (фокусирует поле), читается screen reader. Нажмите на текст «Email адрес» — поле получит фокус.

IX

CSS Архитектура

CSS — не таблица стилей, а язык описания дизайн-системы. Специфичность, токены и скоупинг — это фундамент, а не опциональные практики.

Специфичность Принцип минимальной специфичности — никаких ID в CSS
Плохо — ID-селекторы
/* Специфичность: 1-0-0 */ #header { background: blue; } /* Чтобы переопределить — нужен ещё ID или !important — начинается война */ #header#header { background: red; }
1-0-0 #header
→ трудно переопределить
Хорошо — только классы
/* Специфичность: 0-1-0 */ .header { background: blue; } /* Легко переопределить другим классом */ .header--dark { background: var(--bg-dark); }
0-1-0 .header
→ легко расширять

Порядок специфичности от низкой к высокой: теги (0-0-1) → классы/атрибуты (0-1-0) → ID (1-0-0) → inline styles → !important. Пишите стили с наименьшей необходимой специфичностью.

Скоупинг Все дочерние стили закреплены за корневым элементом компонента
Плохо — глобальные стили
/* Глобальный побочный эффект */ h2 { color: blue; } p { margin: 0; } /* Стиль без корневого якоря */ .title { font-size: 18px; } .content { padding: 16px; }
Хорошо — через корень компонента
/* Все стили через корневой класс */ .card { ... } .card .card__title { font-size: 18px; } .card .card__content { padding: 16px; } /* BEM-модификатор */ .card--featured .card__title { color: var(--color-primary); }
X

Инжиниринг компонента

Инжиниринг — это не только «чтобы работало», но и безопасность, производительность и чистота. Делегирование событий, XSS-защита и батчинг DOM обновлений — обязательные паттерны.

События Делегирование событий — один обработчик на корне вместо N на элементах
Плохо — обработчик на каждом элементе
// N кнопок = N слушателей в памяти document.querySelectorAll('.btn') .forEach(btn => btn.addEventListener('click', handler) ); // Динамически добавленные кнопки — // слушателей нет, нужно добавлять вручную

100 кнопок = 100 слушателей. Динамически добавленные элементы не обрабатываются. Утечка памяти при уничтожении компонента.

Хорошо — один делегированный обработчик
// Один слушатель на корне root.addEventListener('click', e => { const btn = e.target .closest('[data-action]'); if (!btn) return; switch (btn.dataset.action) { case 'submit': ...; break; case 'delete': ...; break; } });

Один слушатель на корне. Работает с динамически добавленными элементами. Удаляется одним abort().

Безопасность XSS — весь пользовательский ввод должен быть экранирован
Плохо — прямая вставка в innerHTML
// Опасно! Пользователь вводит: // <img src=x onerror="alert('XSS')"> const userInput = input.value; el.innerHTML = `<p>${userInput}</p>`; // Скрипт выполнится в браузере!

Любой HTML из пользовательского ввода в innerHTML — это потенциальная XSS-уязвимость. Злоумышленник может выполнить произвольный JS.

Хорошо — экранирование или textContent
// Вариант 1: textContent (только текст) el.textContent = userInput; // Вариант 2: экранирование function esc(s) { return String(s) .replace(/[<>"'&]/g, c => ({'&':'&amp;','<':'&lt;', '>':'&gt;'}[c])); } el.innerHTML = `<p>${esc(userInput)}</p>`;

textContent не интерпретирует HTML. Если нужен HTML — экранируйте все спецсимволы перед вставкой. Любые данные извне — URL, localStorage, API — считаются недоверенными.

Перфоманс Layout thrashing — разделяйте чтение и запись DOM
Плохо — чередование read/write
// Браузер делает reflow на каждой строке el1.style.height = el1.offsetHeight + 'px'; // read→write el2.style.height = el2.offsetHeight + 'px'; // read→write // 2 принудительных reflow подряд
Хорошо — сначала все чтения, потом все записи
// Один reflow: сначала читаем всё const h1 = el1.offsetHeight; const h2 = el2.offsetHeight; // Потом пишем всё el1.style.height = h1 + 'px'; el2.style.height = h2 + 'px';
DOM Пакетные обновления DOM через DocumentFragment
Плохо — множественные вставки в живой DOM
// N вставок = N перерисовок items.forEach(item => list.appendChild(createEl(item)) );
Хорошо — один DocumentFragment
// 1 вставка = 1 перерисовка const frag = document.createDocumentFragment(); items.forEach(item => frag.appendChild(createEl(item)) ); list.replaceChildren(frag);

DocumentFragment — виртуальный контейнер вне DOM. Все изменения накапливаются в нём, а потом вставляются одной операцией. Одна перерисовка вместо N.

Принципы не устаревают, когда меняются фреймворки.
Bootstrap, Tailwind, React, Vue — инструменты для реализации принципов, а не сами принципы.
