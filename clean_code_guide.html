[DRY] — Don't Repeat Yourself

"Every piece of knowledge must have a single, unambiguous, authoritative representation" — Andy Hunt & Dave Thomas

NEVER DUPLICATE LOGIC. REDUNDANCY IS THE ENEMY OF STABILITY.

DRY is not about copy-pasting code — it is about duplicating knowledge. When a business rule changes, you must change it in exactly one place. There are three types of DRY violations: Imposed (the team forces duplication), Inadvertent (developer did not notice), and Impatient (it was faster to copy).

  • Logic duplication — the same algorithm in multiple places
  • Data structure duplication — the same shape described twice
  • Documentation duplication — a comment that restates the code
❌ BAD — order total logic duplicated
<?php
class OrderController
{
    public function create(array $items, int $userId): Order
    {
        $total = 0;
        foreach ($items as $item) {
            $total += $item['price'] * $item['qty'];
        }
        if ($total > 1000) { $total *= 0.95; } // discount rule
        return new Order($userId, $items, $total);
    }

    public function preview(array $items): array
    {
        $total = 0;
        foreach ($items as $item) {
            // SAME LOGIC — change the threshold once,
            // must remember to change it here too
            $total += $item['price'] * $item['qty'];
        }
        if ($total > 1000) { $total *= 0.95; }
        return ['total' => $total, 'count' => count($items)];
    }
}
✅ GOOD — single source of truth
<?php
final class OrderPricer
{
    private const BULK_THRESHOLD = 1000;
    private const BULK_DISCOUNT  = 0.95;

    /** @param array<array{price:float,qty:int}> $items */
    public function compute(array $items): float
    {
        $sub = array_sum(array_map(
            fn($i) => $i['price'] * $i['qty'], $items
        ));
        return $sub > self::BULK_THRESHOLD
            ? $sub * self::BULK_DISCOUNT
            : $sub;
    }
}

class OrderController
{
    public function __construct(
        private readonly OrderPricer $pricer
    ) {}

    public function create(array $items, int $userId): Order
    {
        return new Order($userId, $items,
            $this->pricer->compute($items));
    }

    public function preview(array $items): array
    {
        return [
            'total' => $this->pricer->compute($items),
            'count' => count($items),
        ];
    }
}

Principle

[KISS] — Keep It Simple, Stupid

"Simplicity is the prerequisite for reliability" — Edsger Dijkstra

SIMPLICITY OVER COMPLEXITY. AVOID OVER-ENGINEERING.

Most systems work best if they are kept simple. Complexity is the enemy of understanding, testing, and change. It arrives in two flavors: Essential complexity (inherent to the problem) and Accidental complexity (we created it ourselves). The engineer's job is to eliminate the second type entirely.

  • Simple code is read — complex code is deciphered
  • Simple code is tested with one mock — complex code needs twenty
  • Simple code changes safely — complex code breaks on touch
❌ BAD — a "pricing engine" for one rule
<?php
// 80 lines of pipeline infrastructure to do: price *= 0.95
interface PricingStage {
    public function process(

    ): float;
}

    private array $stages = [];

        $this->stages[] = $s; return $this;

        return array_reduce(
            $this->stages,
            fn($p,$s) => $s->process($p,$ctx,[]),
            $p

    }
}
// ... BulkDiscountStage, TaxStage, RoundingStage ...
// For a blog with one price rule. Never extended.

<?php

        float   $price,

Principle

[YAGNI] — You Aren't Gonna Need It

"Implement things when you actually need them, never when you just foresee that you need them" — Ron Jeffries (XP)

CODE ONLY WHAT IS MANDATED. NO GHOST FEATURES.

Developers are compelled to build "foundations for the future" — universal plugin systems, abstract frameworks, endlessly configurable engines. Over 80% of that speculative code is never used. It only makes the system harder to understand, slower to build, and generates real bugs around imagined requirements.

  • "We might need to support N providers" → Write for one. Refactor when the second actually arrives.
  • Every line of code is maintenance liability
  • Deleting dead code later is harder than never writing it

[NAME] — Naming

"Names in code are the primary form of documentation"

NOUNS FOR DATA, VERBS FOR ACTIONS. BE EXPLICIT.

A good name reveals intention — it requires no comment. The reader must understand what a variable, function, or class does without reading its implementation. Naming rules:

  • Nouns for classes, variables, properties: UserRepository, $orderTotal
  • Verbs for methods and functions: calculateDiscount(), sendEmail()
  • Booleans prefixed with is/has/can/should: $isActive, hasPermission()
  • Collections in plural or with List/Map/Set suffix
  • Name length ∝ scope width: a loop counter = $i, a global flag = $isEmailVerificationRequired

Principle

[UNIT] — Atomic Functions

"A function should do one thing, do it well, and do only that" — Robert C. Martin

KEEP FUNCTIONS ATOMIC. ONE TASK PER SUBROUTINE.

A function does one thing. If you cannot describe it in one sentence without the word "and", it does too much. Metrics:

  • Length: 5–20 lines = healthy, >50 = almost always a problem
  • Abstraction level: all operations inside sit at the same level
  • Arguments: 0–2 = ideal, 3 = acceptable, >4 = introduce a parameter object
  • Side effects: documented, or extracted to a clearly named method
❌ BAD — "God Function" doing 6 things
<?php
function processRegistration(array $data): bool
{
    // 1. Validate
    if (empty($data['email'])
     || !filter_var($data['email'], FILTER_VALIDATE_EMAIL))
        return false;
    if (strlen($data['password']) < 8)
        return false;

    // 2. Hash password
    $hash = password_hash($data['password'],
                PASSWORD_BCRYPT, ['cost' => 12]);

    // 3. Save to DB
    $stmt = $this->pdo->prepare(
        'INSERT INTO users (email,hash,created_at) VALUES(?,?,NOW())'
    );
    $stmt->execute([$data['email'], $hash]);
    $id = $this->pdo->lastInsertId();

    // 4. Send email
    mail($data['email'], 'Welcome!', 'Thanks!');

    // 5. Create profile row
    $this->pdo->exec("INSERT INTO profiles(user_id) VALUES($id)");

    // 6. Log it
    error_log("Registered: {$data['email']}");
    return true;
}
✅ GOOD — orchestration of atomic units
<?php
// Each collaborator does exactly one thing
final class RegisterUserUseCase
{
    public function __construct(
        private readonly RegistrationValidator $validator,
        private readonly PasswordHasher        $hasher,
        private readonly UserRepository        $users,
        private readonly UserMailer            $mailer,
        private readonly LoggerInterface       $log
    ) {}

    public function execute(RegisterCommand $cmd): UserId
    {
        $this->validator->validate($cmd);       // 1. validate
        $hash = $this->hasher->hash(            // 2. hash
            $cmd->password

            email: $cmd->email,
            hash:  $hash
        );

        $this->log->info('User registered',     // 5. log
            ['userId' => $id->value()]
        );

Principle

[INFO] — Readability & Comments

"Write code as if the next maintainer is a violent psychopath who knows where you live"

COMMENTS JUSTIFY INTENT, NOT SYNTAX. CODE MUST BE SELF-DOCUMENTING.

Good code documents itself. Comments answer "Why?" — never "What?". The code already answers "What".

  • Bad comment: restates what the code visibly does
  • Good comment: explains a non-obvious business decision, warns about a trap, references a ticket/RFC
  • Best comment: absent — because the code is clear without it

Five object-oriented design principles by Robert C. Martin. The academic core of any serious OOP course.
SOLID — S

Single Responsibility Principle

A module should have one, and only one, reason to change

ONE MODULE, ONE REASON TO CHANGE.

SRP is commonly misread as "a class should do one thing." The precise statement is: a class should have one reason to change. The "reason" is a stakeholder or business actor who requests changes. If two different actors own a single class, SRP is violated.

❌ BAD — 4 reasons to change one class
<?php
class UserService
{
    // Changes when DB schema changes
    public function save(User $u): void {
        $this->pdo->prepare('INSERT INTO users…')->execute([…]);
    }
    // Changes when email provider changes
    public function sendWelcome(User $u): void {
        mail($u->email, 'Welcome!', '…');
    }
    // Changes when report template changes
    public function generateReport(User $u): string {
        return "<pdf>…{$u->name}…</pdf>";
    }
    // Changes when payment processor changes
    public function charge(User $u): void {
        Stripe::charge($u->paymentMethodId, 999);
    }
}
✅ GOOD — one class, one reason
<?php
// DB team owns this
final class UserRepository {
    public function save(User $u): void { /* … */ }
    public function findById(UserId $id): ?User { /* … */ }
}

    public function sendWelcome(User $u): void { /* … */ }

    public function generate(User $u): Report { /* … */ }

    public function charge(User $u, Plan $p): Receipt { /* … */ }

SRP check: Ask "Who asks us to change this class?" — if there are two different stakeholders, the principle is violated.
SOLID — O

Open / Closed Principle

Open for extension, closed for modification — Bertrand Meyer, 1988

EXTEND SYSTEM VIA NEW CODE, NOT MODIFICATION.

Adding new behavior should mean writing new code, not editing existing code. Every time you open a file to add a new if for a new case, OCP is violated. Achieved through polymorphism, configuration, and extension patterns like Strategy and Decorator.

SOLID — L

Liskov Substitution Principle

Subtypes must be substitutable for their base types — Barbara Liskov, 1987

SUBTYPES MUST BE INTERCHANGEABLE WITHOUT BREAKING CORRECTNESS.

If S is a subtype of T, replacing a T with an S must not break the program. LSP violations:

  • Subclass throws exceptions where the parent does not
  • Subclass makes methods a no-op or silently ignores them
  • Subclass strengthens preconditions or weakens postconditions
  • Classic example: Square extends Rectangle
❌ BAD — Square breaks Rectangle's contract
<?php
class Rectangle {
    protected float $w, $h;
    public function setWidth(float $w):  void { $this->w = $w; }
    public function setHeight(float $h): void { $this->h = $h; }
    public function area(): float { return $this->w * $this->h; }
}
// Square overrides setWidth → changes both dimensions
class Square extends Rectangle {
    public function setWidth(float $w): void {
        $this->w = $this->h = $w; // contract broken
    }
    public function setHeight(float $h): void {
        $this->w = $this->h = $h;
    }
}
function testArea(Rectangle $r): void {
    $r->setWidth(5);
    $r->setHeight(4);
    assert($r->area() === 20.0); // FAILS for Square: 16 ≠ 20
}
testArea(new Square()); // ← violates LSP
✅ GOOD — separate types, shared interface
<?php
interface HasArea {
    public function area(): float;
}
final class Rectangle implements HasArea {
    public function __construct(
        private readonly float $width,
        private readonly float $height
    ) {}
    public function area(): float {
        return $this->width * $this->height;
    }
}
final class Square implements HasArea {
    public function __construct(
        private readonly float $side
    ) {}
    public function area(): float {
        return $this->side ** 2;
    }
}
// Any HasArea works correctly — true substitution
function totalArea(HasArea ...$shapes): float {
    return array_sum(array_map(fn($s) => $s->area(), $shapes));
}
// 20 + 9 + 12 = 41 — fully predictable
$total = totalArea(new Rectangle(5,4), new Square(3), new Rectangle(2,6));
SOLID — I

Interface Segregation Principle

Clients should not be forced to depend on interfaces they do not use

PREFER GRANULAR ROLE INTERFACES OVER BULK OBJECTS.

A "fat interface" has too many methods. Classes are forced to implement (or stub) things they don't need. The tell-tale sign: throw new NotImplementedException() in a required method. Split fat interfaces into focused role interfaces.

❌ BAD — fat interface forces stubs
<?php
interface Worker {
    public function work(): void;
    public function eat(): void;
    public function sleep(): void;
    public function pay(float $salary): void;
    public function manageTeam(array $team): void;
}
// Robot can work and be paid, but can't eat or sleep
class Robot implements Worker {
    public function work(): void { /* real logic */ }
    public function pay(float $s): void { /* ok */ }
    // Forced to stub — violates LSP too
    public function eat(): void {
        throw new \LogicException('Robots do not eat');

<?php
declare(strict_types=1);
namespace Domain\Order;

// This class knows NOTHING about Laravel, MySQL, or HTTP.
final class Order
{
    private OrderStatus $status;
    /** @var OrderItem[] */  private array $items        = [];
    /** @var DomainEvent[] */ private array $domainEvents = [];

    private function __construct(
        private readonly OrderId    $id,
        private readonly CustomerId $customerId,
    ) {
        $this->status = OrderStatus::Draft;
    }

    public static function create(OrderId $id, CustomerId $customerId): self
    {
        $o = new self($id, $customerId);
        $o->domainEvents[] = new OrderCreated($id, $customerId);
        return $o;
    }

    public function addItem(ProductId $pid, int $qty, Money $price): void
    {
        if ($this->status !== OrderStatus::Draft) {
            throw new \DomainException('Only draft orders can be modified');
        }
        $this->items[] = new OrderItem($pid, $qty, $price);
    }

    public function place(): void
    {
        if (empty($this->items)) {
            throw new \DomainException('Order must have at least one item');
        }
        $this->status = OrderStatus::Placed;
        $this->domainEvents[] = new OrderPlaced($this->id, $this->getTotal());
    }

    public function getTotal(): Money
    {
        return array_reduce(
            $this->items,
            fn(Money $sum, OrderItem $i) => $sum->add($i->subtotal()),
            Money::zero('USD')
        );
    }

    /** Collect and clear — outbox pattern */
    public function pullDomainEvents(): array
    {
        $events = $this->domainEvents;
        $this->domainEvents = [];
        return $events;
    }
}

final class MySQLOrderRepository implements OrderRepository
{
    public function __construct(
        private readonly \PDO        $pdo,
        private readonly OrderMapper $mapper
    ) {}

    public function findById(OrderId $id): ?Order
    {
        $stmt = $this->pdo->prepare(
            'SELECT o.*, oi.* FROM orders o
             JOIN order_items oi ON oi.order_id = o.id
             WHERE o.id = :id'
        );
        $stmt->execute(['id' => $id->value()]);
        $rows = $stmt->fetchAll(\PDO::FETCH_ASSOC);
        return empty($rows) ? null : $this->mapper->toDomain($rows);
    }

    public function save(Order $order): void
    {
        $this->pdo->beginTransaction();
        try {
            $data = $this->mapper->toPersistence($order);
            $this->upsert($data['order']);
            $this->replaceItems($order->id(), $data['items']);
            $this->pdo->commit();
        } catch (\Throwable $e) {
            $this->pdo->rollBack();
            throw new PersistenceException('Failed to save order', 0, $e);
        }
    }
}

Define an interface for creating an object, but let subclasses decide which class to instantiate

Problem: You need to create objects but the exact type is determined at runtime. Hard-coding new ConcreteClass() couples caller to implementation.
Use when: The class to instantiate isn't known until runtime; subclasses or configuration should decide the type; you want to centralize object creation logic.

JavaScript — Builder (SQL query builder)
class QueryBuilder {
    #table  = null;
    #wheres = [];
    #select = ['*'];
    #limit  = null;
    #order  = null;

    from(table)          { this.#table = table;        return this; }
    select(...cols)      { this.#select = cols;         return this; }
    where(col, op, val)  { this.#wheres.push({col,op,val}); return this; }
    orderBy(col, dir='ASC') { this.#order = `${col} ${dir}`; return this; }
    limit(n)             { this.#limit = n;             return this; }

    build() {
        if (!this.#table) throw new Error('from() is required');
        let sql = `SELECT ${this.#select.join(', ')} FROM ${this.#table}`;
        if (this.#wheres.length) {
            const clauses = this.#wheres.map(w => `${w.col} ${w.op} ?`);
            sql += ` WHERE ${clauses.join(' AND ')}`;
        }
        if (this.#order)  sql += ` ORDER BY ${this.#order}`;
        if (this.#limit)  sql += ` LIMIT ${this.#limit}`;
        const params = this.#wheres.map(w => w.val);
        return { sql, params };
    }
}

Decorator

Add responsibilities to an object dynamically by wrapping it — without altering the original class

ADD WRAPPERS (LOGGING / CACHE) WITHOUT ALTERING SOURCE.

Problem: You need to add cross-cutting concerns (caching, logging, metrics, retry) to objects at runtime. Inheritance explodes into combinatorial subclass hell. Decorator wraps the same interface, stacking behaviors like layers.
Key rule: Decorator and the component implement the same interface.

PHP — Cache with Decorator stack (Logging + Metrics + Redis)
<?php
interface CacheInterface {
    public function get(string $key): mixed;
    public function set(string $key, mixed $value, int $ttl): void;
    public function delete(string $key): void;
}

// Real implementation — the innermost layer
final class RedisCache implements CacheInterface {
    public function __construct(private \Redis $redis) {}
    public function get(string $key): mixed  { return $this->redis->get($key) ?: null; }
    public function set(string $key, mixed $v, int $ttl): void { $this->redis->setex($key, $ttl, serialize($v)); }
    public function delete(string $key): void { $this->redis->del($key); }
}

// Decorator #1 — adds logging, knows nothing about Redis
final class LoggingCache implements CacheInterface {
    public function __construct(
        private readonly CacheInterface  $inner,
        private readonly LoggerInterface $logger
    ) {}
    public function get(string $key): mixed {
        $value = $this->inner->get($key);
        $this->logger->debug('Cache ' . ($value !== null ? 'HIT' : 'MISS'), ['key' => $key]);
        return $value;
    }
    public function set(string $key, mixed $v, int $ttl): void {
        $this->inner->set($key, $v, $ttl);
        $this->logger->debug('Cache SET', ['key' => $key, 'ttl' => $ttl]);
    }
    public function delete(string $key): void {
        $this->inner->delete($key);
        $this->logger->debug('Cache DELETE', ['key' => $key]);
    }
}

// Decorator #2 — adds metrics tracking
final class MetricsCache implements CacheInterface {
    private int $hits = 0, $misses = 0;
    public function __construct(
        private readonly CacheInterface  $inner,
        private readonly Metrics         $metrics
    ) {}
    public function get(string $key): mixed {
        $value = $this->inner->get($key);
        $value !== null ? $this->hits++ : $this->misses++;
        $this->metrics->gauge('cache.hit_rate', $this->hits / max(1, $this->hits + $this->misses));
        return $value;
    }
    public function set(string $key, mixed $v, int $ttl): void { $this->inner->set($key,$v,$ttl); }
    public function delete(string $key): void { $this->inner->delete($key); }
}

// Assemble the stack — innermost first, outermost last

    new LoggingCache(
        new RedisCache($redis),
        $logger
    ),
    $metrics
);
// All consumers see: CacheInterface. Stack is transparent.

            try {
                return await fetchFn(url, options);
            } catch (err) {
                if (attempt === retries) throw err;
                await new Promise(r => setTimeout(r, delayMs * attempt));
            }
        }
    };

        const res = await fetchFn(url, options);
        logger.info(`← ${url} ${res.status} (${Date.now() - t0}ms)`);
        return res;
    };
}

<?php
interface SortStrategy {
    /** @param mixed[] $data */
    public function sort(array &$data): void;
}
class BubbleSort implements SortStrategy {
    public function sort(array &$data): void {
        $n = count($data);
        for ($i = 0; $i < $n - 1; $i++)
            for ($j = 0; $j < $n - $i - 1; $j++)
                if ($data[$j] > $data[$j+1])
                    [$data[$j], $data[$j+1]] = [$data[$j+1], $data[$j]];
    }
}

    public function sort(array &$data): void { /* quicksort */ }

        $this->strategy = $s;
    }
    public function sort(array $data): array {
        $this->strategy->sort($data);
        return $data;
    }
}

    #handlers = new Map();

        if (!this.#handlers.has(event)) this.#handlers.set(event, []);
        this.#handlers.get(event).push(handler);
        return () => this.off(event, handler); // returns unsubscribe fn
    }

        const list = this.#handlers.get(event) ?? [];
        this.#handlers.set(event, list.filter(h => h !== handler));
    }

        (this.#handlers.get(event) ?? []).forEach(h => h(payload));
    }
}

Command

Encapsulate a request as an object — enabling undo, queuing, and logging

Problem: You need to parameterize actions, queue them, log them, or support undo. Command turns requests into standalone objects with a unified execute() interface.

PHP — Command pattern with undo support
<?php
interface Command {
    public function execute(): void;
    public function undo(): void;
}

final class MoveFileCommand implements Command
{
    private ?string $previousPath = null;

    public function __construct(
        private readonly Filesystem $fs,
        private readonly string     $from,
        private readonly string     $to
    ) {}

    public function execute(): void {
        $this->previousPath = $this->from;
        $this->fs->move($this->from, $this->to);
    }

    public function undo(): void {
        if ($this->previousPath === null) return;
        $this->fs->move($this->to, $this->previousPath);
    }
}

final class DeleteFileCommand implements Command
{
    private ?string $backup = null;

    public function __construct(
        private readonly Filesystem $fs,
        private readonly string     $path
    ) {}

    public function execute(): void {
        $this->backup = $this->fs->readContents($this->path);
        $this->fs->delete($this->path);
    }

    public function undo(): void {
        if ($this->backup !== null) {
            $this->fs->write($this->path, $this->backup);
        }
    }
}

// Command history — supports multi-level undo
final class CommandHistory
{
    /** @var Command[] */
    private array $history = [];

    public function execute(Command $cmd): void {
        $cmd->execute();
        $this->history[] = $cmd;
    }

    public function undo(): void {
        $cmd = array_pop($this->history);
        $cmd?->undo();
    }
}

Template Method

Define the skeleton of an algorithm in a base class; let subclasses fill in specific steps

PHP — Template Method (data import pipeline)
<?php
// Abstract base — defines the invariant algorithm skeleton
abstract class DataImporter
{
    // Template method — the skeleton, cannot be overridden
    final public function import(string $source): ImportResult
    {
        $this->validate($source);        // step 1
        $raw   = $this->read($source);   // step 2 — subclass defines
        $data  = $this->transform($raw); // step 3 — subclass defines
        $count = $this->persist($data);  // step 4
        $this->cleanup($source);         // step 5 (optional hook)
        return new ImportResult($count);
    }

    protected function validate(string $source): void
    {
        if (!file_exists($source)) {
            throw new \RuntimeException("Source not found: $source");
        }
    }

    abstract protected function read(string $source): array;
    abstract protected function transform(array $raw): array;

    protected function persist(array $data): int
    {
        // Default: batch insert into DB
        return $this->db->batchInsert($this->table(), $data);
    }

    protected function cleanup(string $source): void {} // optional hook

    abstract protected function table(): string;
}

// CSV implementation — only fills in CSV-specific steps

{
    protected function read(string $source): array {

Chain of Responsibility

Pass a request along a chain of handlers until one processes it

Use when: Multiple objects may handle a request; the handler isn't known in advance; you want to build middleware pipelines, validation chains, or approval workflows.

PHP — HTTP middleware pipeline
<?php
interface Middleware {
    public function handle(Request $req, callable $next): Response;
}

// Each middleware does its job, then calls $next or short-circuits
final class AuthenticationMiddleware implements Middleware {
    public function handle(Request $req, callable $next): Response {
        if (!$req->hasValidToken()) {
            return Response::unauthorized();
        }
        return $next($req);
    }
}
final class RateLimitMiddleware implements Middleware {
    public function __construct(private readonly RateLimiter $limiter) {}
    public function handle(Request $req, callable $next): Response {
        if ($this->limiter->isExceeded($req->ip())) {
            return Response::tooManyRequests();
        }
        return $next($req);
    }
}
final class LoggingMiddleware implements Middleware {
    public function __construct(private readonly LoggerInterface $log) {}
    public function handle(Request $req, callable $next): Response {
        $this->log->info('→ ' . $req->path());
        $response = $next($req);
        $this->log->info('← ' . $response->status());
        return $response;
    }
}

// Pipeline builder
final class Pipeline {
    /** @var Middleware[] */
    private array $middlewares = [];
    public function pipe(Middleware $m): static { $this->middlewares[] = $m; return $this; }

    public function run(Request $req, callable $final): Response {
        $chain = array_reduce(
            array_reverse($this->middlewares),
            fn($carry, $m) => fn($r) => $m->handle($r, $carry),
            $final
        );
        return $chain($req);
    }
}

$response = (new Pipeline())
    ->pipe(new LoggingMiddleware($logger))
    ->pipe(new AuthenticationMiddleware())
    ->pipe(new RateLimitMiddleware($limiter))
    ->run($request, fn($req) => $controller->handle($req));

Mediator

Reduce direct dependencies between components by routing communication through a central hub

Problem: Components communicate directly with each other — a web of M×N connections that becomes unmaintainable. Mediator (often a Command Bus or Message Bus) routes messages centrally, keeping components decoupled.

<?php
// Command Bus — routes commands to their handlers
final class CommandBus
{
    /** @var array<class-string, callable> */
    private array $handlers = [];

        $this->handlers[$commandClass] = $handler;

final class MySQLUserRepository implements UserRepository
{
    public function __construct(private readonly \PDO $pdo) {}

    public function findById(UserId $id): ?User {
        $stmt = $this->pdo->prepare('SELECT * FROM users WHERE id = :id AND deleted_at IS NULL');
        $stmt->execute(['id' => $id->value()]);
        $row = $stmt->fetch(\PDO::FETCH_ASSOC);
        return $row ? User::fromArray($row) : null;
    }

    public function findByEmail(Email $email): ?User {
        $stmt = $this->pdo->prepare('SELECT * FROM users WHERE email = :email');
        $stmt->execute(['email' => $email->value()]);
        $row = $stmt->fetch(\PDO::FETCH_ASSOC);
        return $row ? User::fromArray($row) : null;
    }

    public function findActiveByRole(Role $role): array {
        $stmt = $this->pdo->prepare(
            'SELECT * FROM users WHERE role = :role AND status = "active" AND deleted_at IS NULL'
        );
        $stmt->execute(['role' => $role->value]);
        return array_map(fn($r) => User::fromArray($r), $stmt->fetchAll(\PDO::FETCH_ASSOC));
    }

    public function save(User $user): void {
        $this->pdo->prepare(
            'INSERT INTO users (id, email, name, role, status, created_at)
             VALUES (:id, :email, :name, :role, :status, :created_at)
             ON DUPLICATE KEY UPDATE email=:email, name=:name, role=:role, status=:status'
        )->execute([
            'id'         => $user->id()->value(),
            'email'      => $user->email()->value(),
            'name'       => $user->name(),
            'role'       => $user->role()->value,
            'status'     => $user->status()->value,
            'created_at' => $user->createdAt()->format('Y-m-d H:i:s'),
        ]);
    }

    public function delete(UserId $id): void {
        // Soft delete
        $this->pdo->prepare('UPDATE users SET deleted_at = NOW() WHERE id = :id')

    }

Low-Level Engineering Standards
Four non-negotiable disciplines that keep code contracts explicit, failures visible, and external data safe. These are the difference between production code and scripts.
Standard

[TYPE] — Strict Typing

Contracts between components must be machine-enforced, not convention-based

STRICT TYPE HINTING. CONTRACTS MUST BE ENFORCED AT THE BOUNDARY.

Types are executable documentation. A string parameter says nothing — a Email value object says exactly what it accepts and rejects. Untyped code pushes failures from compile-time (zero cost) to runtime in production (high cost). The richer the type system you use, the less validation logic you need to write.

  • Primitive obsession: string $email lets any string through; Email $email guarantees a valid email
  • Value Objects: immutable, self-validating wrappers around primitives
  • Return types: make null-returns explicit with ?Type or Option patterns
  • Never use mixed, array, or object where a specific type is possible

[LOGS] — Structured Logging

A system that cannot observe itself cannot be operated in production

SYSTEM LOGGING ONLY. PRINT IS FOR DEBUGGING, NOT PRODUCTION.

Production logging has three rules: it must be structured (machine-parseable, not freeform strings), it must carry context (trace IDs, user IDs, correlation data), and it must use appropriate levels. A log entry is a fact — not a debug breadcrumb left in a hurry.

  • DEBUG — detailed trace, off in production
  • INFO — normal operations: request handled, job completed
  • WARNING — unexpected but recoverable: retry attempt, fallback used
  • ERROR — failure requiring investigation, service still running
  • CRITICAL — service degraded or data integrity at risk

[EXCEPT] — Custom Exception Hierarchy

Generic exceptions are noise — domain exceptions are signal

CUSTOM ERROR CLASSES FOR BUSINESS-SPECIFIC FAILURES.

A RuntimeException tells the caller almost nothing. A PaymentDeclinedException tells it exactly what went wrong, whether to retry, and which HTTP status to return. Your exception hierarchy is part of your API contract — design it deliberately.

  • Exceptions are part of the domain — live next to the domain code
  • Catch specific exceptions, never bare catch (\Exception $e) unless at the app boundary
  • Exceptions carry context: the ID that failed, the value that was invalid
  • Mark exceptions as retryable or not for automatic retry policies

[STATE] — Externalized State

A process that holds critical state in memory is one crash away from data loss

STATE IS EXTERNALIZED. TRACK OFFSET, STATUS, AND CHECKSUMS.

Any state that must survive a process restart — job progress, cursor positions, processing offsets, transaction status — must live outside the process in durable storage. In-memory state is ephemeral. The process must be able to restart cold and reconstruct exactly where it was.

  • Checkpoint — the last verified consistent state
  • Checksum — proof that the data we read is what was written
  • Idempotency key — proof that we haven't already processed this item

Resilience

[RESUME] — Auto-Recovery & Checkpoints

A system that requires manual intervention after every crash is not production-ready

AUTO-RECOVERY. RESUME FROM LAST CHECKPOINT AFTER POWER LOSS.

Resilient processes detect their own state on startup and resume automatically from the last consistent checkpoint. This requires three things: checkpoints written before progress is lost, a startup routine that reads the checkpoint, and idempotent operations so re-processing a recovered segment causes no harm.

  • Checkpoint before acknowledging a message, not after
  • Use exponential backoff for transient failures
  • Set a dead-letter threshold: after N retries, move to error queue
  • Log every retry attempt with cause — silent retries hide bugs
PHP — self-recovering queue worker with exponential backoff
<?php
final class ResilientWorker
{
    private const MAX_RETRIES  = 5;
    private const BASE_DELAY   = 1;   // seconds
    private const MAX_DELAY    = 60;  // seconds

    public function __construct(
        private readonly Queue             $queue,
        private readonly MessageProcessor  $processor,
        private readonly JobStateRepository $state,
        private readonly LoggerInterface   $logger,
        private readonly DeadLetterQueue   $dlq,
    ) {}

    public function boot(): void
    {
        // ── Self-diagnosis on startup ──────────────────────────
        $this->runHealthChecks();

        // ── Recover any in-progress jobs from previous run ───
        $this->recoverInterruptedJobs();

        // ── Start processing loop ─────────────────────────────
        $this->process();
    }

    private function recoverInterruptedJobs(): void
    {
        $stuckJobs = $this->state->findByStatus('running');

        foreach ($stuckJobs as $job) {
            $this->logger->warning('Recovering interrupted job', [
                'job_id' => $job->id,
                'offset' => $job->offset,
            ]);
            // Mark as pending so main loop picks them up
            $this->state->updateStatus($job->id, 'pending');
        }
    }

    private function process(): void
    {
        while (true) {
            $message = $this->queue->pop();
            if ($message === null) { sleep(1); continue; }

            $this->handleWithRetry($message);
        }
    }

    private function handleWithRetry(Message $message): void
    {
        $attempts = $message->attempts();

        try {
            $this->state->updateStatus($message->id(), 'running');
            $this->processor->handle($message);
            $this->state->updateStatus($message->id(), 'completed');
            $this->queue->ack($message);

        } catch (TransientException $e) {
            // Retryable — exponential backoff
            if ($attempts >= self::MAX_RETRIES) {
                $this->logger->error('Max retries exceeded, moving to DLQ', [
                    'message_id' => $message->id(),
                    'attempts'   => $attempts,
                    'last_error' => $e->getMessage(),
                ]);
                $this->dlq->push($message->withError($e));
                $this->queue->ack($message);
                return;
            }

            $delay = min(self::BASE_DELAY * (2 ** $attempts), self::MAX_DELAY);
            $this->logger->warning('Transient error, scheduling retry', [
                'message_id' => $message->id(),
                'attempt'    => $attempts + 1,
                'retry_in'   => $delay,
                'error'      => $e->getMessage(),
            ]);
            $this->queue->nackWithDelay($message, $delay);

        } catch (PermanentException $e) {
            // Non-retryable — straight to DLQ
            $this->logger->error('Permanent error, moving to DLQ', [
                'message_id' => $message->id(),
                'error'      => $e->getMessage(),
            ]);
            $this->dlq->push($message->withError($e));
            $this->queue->ack($message);
        }
    }

    private function runHealthChecks(): void
    {
        // Implemented in [HEALTH] section below
    }
}
JavaScript — retry with exponential backoff utility
/**
 * Execute fn with automatic retry on transient failures.
 * @param {Function} fn          - Async function to execute
 * @param {Object}   options
 * @param {number}   options.maxRetries   - Maximum retry attempts (default: 3)
 * @param {number}   options.baseDelayMs  - Base delay in ms (default: 500)
 * @param {number}   options.maxDelayMs   - Cap on delay (default: 30000)
 * @param {Function} options.isRetryable  - Predicate: should we retry this error?
 */
async function withRetry(fn, {
    maxRetries  = 3,
    baseDelayMs = 500,
    maxDelayMs  = 30_000,
    isRetryable = (err) => err.retryable !== false,
    onRetry     = (err, attempt) => {},
} = {}) {
    let lastError;

    for (let attempt = 1; attempt <= maxRetries + 1; attempt++) {
        try {
            return await fn(attempt);
        } catch (err) {
            lastError = err;

            if (attempt > maxRetries || !isRetryable(err)) {
                throw err;
            }

            // Exponential backoff with jitter
            const exponential = baseDelayMs * Math.pow(2, attempt - 1);
            const jitter       = Math.random() * baseDelayMs;
            const delay        = Math.min(exponential + jitter, maxDelayMs);

            onRetry(err, attempt);
            await new Promise(r => setTimeout(r, delay));
        }
    }
    throw lastError;
}

[ATOM] — Atomic I/O

A file either exists in its complete form or not at all — there is no in-between

ATOMIC I/O. WRITE TO TMP, THEN SWAP. NO CORRUPT FILES EVER.

Direct writes are not atomic. If the process crashes between the first and last byte of a write, you get a truncated, corrupt file. The write-to-temp-then-rename pattern is atomic on all major filesystems: the rename is a single syscall, it either completes or doesn't. The reader always sees the old complete file or the new complete file — never a partial one.

  • Write to file.tmp → fsync to flush OS buffer → rename to file
  • rename() is atomic on POSIX — guaranteed by the filesystem
  • The same principle applies to database transactions: begin → write → commit
  • Use transactions for multi-table writes — partial writes corrupt relational data

<?php
function saveConfig(array $config, string $path): void
{
    $tmpPath = $path . '.tmp.' . getmypid();
    $content = json_encode($config, JSON_PRETTY_PRINT | JSON_THROW_ON_ERROR);

    // 1. Write to temp file
    $bytes = file_put_contents($tmpPath, $content, LOCK_EX);
    if ($bytes === false) {
        throw new \RuntimeException("Write to temp failed: {$tmpPath}");
    }

    // 2. Force OS buffer flush to physical disk
    $fp = fopen($tmpPath, 'r');
    fsync($fp); // PHP 8.1+
    fclose($fp);

    // 3. Atomic rename — this is the commit point
    if (!rename($tmpPath, $path)) {
        unlink($tmpPath);
        throw new \RuntimeException("Atomic rename failed: {$path}");
    }
    // Readers see either old complete file or new complete file.
    // Never a partial write.
}

final class OrderPersister
{
    public function __construct(private readonly \PDO $pdo) {}

    public function save(Order $order, Payment $payment): void

        $this->pdo->beginTransaction();

            // 1. Write order
            $this->pdo->prepare(
                'INSERT INTO orders (id, customer_id, status, total)
                 VALUES (:id, :cid, :status, :total)
                 ON DUPLICATE KEY UPDATE status=:status'
            )->execute([
                'id'     => $order->id(),
                'cid'    => $order->customerId(),
                'status' => $order->status(),
                'total'  => $order->total(),
            ]);

            // 2. Write order items
            foreach ($order->items() as $item) {
                $this->pdo->prepare(
                    'INSERT INTO order_items (order_id, product_id, qty, price)
                     VALUES (:oid, :pid, :qty, :price)'
                )->execute([
                    'oid'   => $order->id(),
                    'pid'   => $item->productId(),
                    'qty'   => $item->quantity(),
                    'price' => $item->unitPrice(),
                ]);
            }

            // 3. Write payment record
            $this->pdo->prepare(
                'INSERT INTO payments (id, order_id, amount, status, gateway)
                 VALUES (:id, :oid, :amount, :status, :gw)'
            )->execute([
                'id'     => $payment->id(),
                'oid'    => $order->id(),
                'amount' => $payment->amount(),
                'status' => $payment->status(),
                'gw'     => $payment->gateway(),
            ]);

            // 4. All succeeded → commit is the atomic "rename"
            $this->pdo->commit();

        } catch (\Throwable $e) {
            // Any failure → undo everything written above
            $this->pdo->rollBack();
            throw new PersistenceException(

[IDEM] — Idempotency

Re-running an operation any number of times must produce the same result as running it once

IDEMPOTENCY. RE-RUNNING A TASK PRODUCES THE SAME RESULT.

Networks fail. Clients time out and retry. Queue messages are delivered more than once. If your operations are not idempotent, retries cause duplicate charges, double emails, corrupted counters. Idempotency is not optional for any system that retries — it is a hard requirement.

  • Idempotency key: a unique ID supplied by the caller, stored server-side — "if you've seen this key, ignore the request"
  • Natural idempotency: SET x = 5 is idempotent; INCREMENT x is not
  • Upsert over insert: INSERT … ON DUPLICATE KEY UPDATE instead of bare INSERT
  • Check-then-act: query for existing state before mutating

final class OrderPlacedConsumer
{

        private readonly ProcessedMessageLog $log,
        private readonly InventoryService    $inventory,
        private readonly WarehouseNotifier   $warehouse,
        private readonly LoggerInterface     $logger,

    public function consume(Message $message): void
    {
        $messageId = $message->id();

        // Guard: have we already processed this exact message?

            $this->logger->info('Skipping duplicate message', [
                'message_id' => $messageId,
            ]);

        // Mark as in-progress (prevents parallel duplicate processing)
        $this->log->markInProgress($messageId);

            // Mark complete — this is our commit point
            $this->log->markProcessed($messageId);

        } catch (\Throwable $e) {
            $this->log->markFailed($messageId, $e->getMessage());
            throw $e; // re-throw so queue can retry
        }
    }
}

class IdempotentHandler {
    constructor(redisClient, ttlSeconds = 86400) {
        this.redis      = redisClient;
        this.ttlSeconds = ttlSeconds;
    }

    // Wrap any handler to make it idempotent
    wrap(handler) {
        return async (req, res) => {
            const key = req.headers['idempotency-key'];
            if (!key) return res.status(400).json({ error: 'Idempotency-Key required' });

            const storeKey = `idem:${key}`;

            // Check for existing response
            const cached = await this.redis.get(storeKey);
            if (cached) {
                const { statusCode, body } = JSON.parse(cached);
                return res.status(statusCode).json({ ...body, replayed: true });
            }

            // Lock: SET NX (only if not exists) — prevents race condition
            const locked = await this.redis.set(storeKey + ':lock', '1',
                { NX: true, EX: 30 }
            );
            if (!locked) return res.status(409).json({ error: 'Concurrent request in progress' });

            try {
                // Capture response
                let capturedStatus = 200;
                let capturedBody   = null;
                const fakeRes = {
                    status: (code) => ({ json: (body) => {
                        capturedStatus = code;
                        capturedBody   = body;
                    }}),
                };

                await handler(req, fakeRes);

                // Store response for future replays
                await this.redis.set(storeKey,
                    JSON.stringify({ statusCode: capturedStatus, body: capturedBody }),
                    { EX: this.ttlSeconds }
                );

                return res.status(capturedStatus).json(capturedBody);
            } finally {
                await this.redis.del(storeKey + ':lock');
            }
        };
    }
}

const idempotent = new IdempotentHandler(redis);

    res.status(201).json({ chargeId: charge.id });
}));

Resilience

[HEALTH] — Self-Diagnosis on Boot

A process must verify that all its dependencies are reachable before accepting work

SELF-DIAGNOSIS ON BOOT. VERIFY INTEGRITY BEFORE EXECUTION.

A process that boots successfully but silently fails on first request is worse than a process that refuses to start. Health checks run on startup (liveness) and periodically (readiness), asserting that every dependency — database, cache, queue, external API — is available and responsive. Kubernetes, Docker, and load balancers use these probes to route traffic only to healthy instances.

  • Liveness probe: is the process alive? Can it respond at all?
  • Readiness probe: is the process ready to accept requests? DB connected?
  • Startup probe: slow-starting processes — don't kill them during warm-up
  • A failing readiness probe removes the instance from the load balancer without killing it
PHP — comprehensive health check system
<?php
interface HealthCheck
{
    public function name(): string;
    public function check(): HealthResult;
}

final class HealthResult
{
    private function __construct(
        public readonly bool   $healthy,
        public readonly string $message,
        public readonly array  $details = [],
        public readonly float  $durationMs = 0.0,
    ) {}

    public static function ok(string $msg = 'OK', array $details = [], float $ms = 0): self {
        return new self(true, $msg, $details, $ms);
    }
    public static function fail(string $msg, array $details = [], float $ms = 0): self {
        return new self(false, $msg, $details, $ms);
    }
}

// Database health check
final class DatabaseHealthCheck implements HealthCheck
{
    public function __construct(private readonly \PDO $pdo) {}

    public function name(): string { return 'database'; }

    public function check(): HealthResult
    {
        $start = microtime(true);
        try {
            $this->pdo->query('SELECT 1');
            return HealthResult::ok('Connected', [], (microtime(true) - $start) * 1000);
        } catch (\PDOException $e) {
            return HealthResult::fail("Connection failed: {$e->getMessage()}",
                [], (microtime(true) - $start) * 1000);
        }
    }
}

// Redis health check
final class RedisHealthCheck implements HealthCheck
{
    public function __construct(private readonly \Redis $redis) {}

    public function name(): string { return 'redis'; }

    public function check(): HealthResult
    {
        $start = microtime(true);
        try {
            $key = 'health:ping:' . uniqid();
            $this->redis->setex($key, 5, 'pong');
            $val = $this->redis->get($key);
            $this->redis->del($key);

            if ($val !== 'pong') {
                return HealthResult::fail('Round-trip failed: unexpected value');
            }
            return HealthResult::ok('Connected', [], (microtime(true) - $start) * 1000);
        } catch (\RedisException $e) {
            return HealthResult::fail("Connection failed: {$e->getMessage()}");
        }
    }
}

// Disk space health check
final class DiskSpaceHealthCheck implements HealthCheck
{
    public function __construct(
        private readonly string $path          = '/',
        private readonly int    $minFreePercent = 10
    ) {}

    public function name(): string { return 'disk_space'; }

    public function check(): HealthResult
    {
        $free  = disk_free_space($this->path);
        $total = disk_total_space($this->path);
        $pct   = ($free / $total) * 100;

        if ($pct < $this->minFreePercent) {
            return HealthResult::fail(
                "Low disk space: {$pct}% free (threshold: {$this->minFreePercent}%)",
                ['free_gb' => round($free / 1e9, 2), 'total_gb' => round($total / 1e9, 2)]
            );
        }
        return HealthResult::ok('OK', [
            'free_gb'  => round($free / 1e9, 2),
            'free_pct' => round($pct, 1),
        ]);
    }
}

// Health Check Runner — aggregates all checks
final class HealthCheckRunner
{
    /** @var HealthCheck[] */
    private array $checks = [];

    public function register(HealthCheck $check): void {
        $this->checks[] = $check;
    }

    public function run(): HealthReport
    {
        $results  = [];
        $allGood  = true;

        foreach ($this->checks as $check) {
            $result            = $check->check();
            $results[$check->name()] = $result;
            if (!$result->healthy) $allGood = false;
        }

        return new HealthReport($allGood, $results);
    }
}

// HTTP health endpoint — for Kubernetes probes
final class HealthController
{
    public function __construct(
        private readonly HealthCheckRunner $runner
    ) {}

    public function readiness(): JsonResponse
    {
        $report = $this->runner->run();
        $status = $report->isHealthy() ? 200 : 503;

        return new JsonResponse([
            'status'  => $report->isHealthy() ? 'healthy' : 'degraded',
            'checks'  => array_map(fn($r) => [
                'healthy'     => $r->healthy,
                'message'     => $r->message,
                'duration_ms' => round($r->durationMs, 2),
                'details'     => $r->details,
            ], $report->results()),
            'timestamp' => date('c'),
        ], $status);
    }

    public function liveness(): JsonResponse
    {
        // Liveness: is the process alive? Always 200 unless process is hanging
        return new JsonResponse(['status' => 'alive'], 200);
    }
}

            ]);
        }
    }

}

class HealthChecker {
    #checks = new Map();

    register(name, checkFn) {
        this.#checks.set(name, checkFn);
        return this;
    }

    async runAll() {
        const results  = {};
        let   allGood  = true;

        await Promise.all([...this.#checks.entries()].map(async ([name, fn]) => {
            const start = Date.now();
            try {
                const details = await Promise.race([
                    fn(),
                    new Promise((_, rej) => setTimeout(() => rej(new Error('Timeout')), 5000)),
                ]);
                results[name] = { healthy: true,  details, durationMs: Date.now() - start };
            } catch (err) {
                results[name] = { healthy: false, error: err.message, durationMs: Date.now() - start };
                allGood = false;
            }
        }));

        return { healthy: allGood, results };
    }
}

    const report = await health.runAll();

    if (!report.healthy) {
        console.error('Startup health checks FAILED:');
        Object.entries(report.results)
            .filter(([, r]) => !r.healthy)
            .forEach(([name, r]) => console.error(`  ✗ ${name}: ${r.error}`));
