#### 28.3.3 Production Readiness Review

##### Purpose

`Production Readiness Review` belongs to the `Reliability Programs` chapter of `Site Reliability Engineering`. In this context, it is about making expected service behavior and recovery responsibilities explicit. The practical goal is to help engineers make a decision that can survive code review, design review, production operation, and later change.

This topic matters because `Site Reliability Engineering` is concerned with balancing feature velocity with explicit reliability targets and operational discipline. A good treatment of `Production Readiness Review` should clarify SLOs, error budgets, incidents, and automation, protect the invariant that the system should have a known user impact, owner, signal, and recovery path for important failures, and give the team observable signals such as budget-aware decisions, tested runbooks, and reduced toil.

##### Core Ideas

- Start with intent. State why `Production Readiness Review` is relevant to the current problem before naming a mechanism.
- Use the chapter boundary. `Reliability Programs` supplies the nearby vocabulary, constraints, and failure modes.
- Protect the invariant: the system should have a known user impact, owner, signal, and recovery path for important failures.
- Use this topic when the team must decide how much failure is acceptable and how to respond when it occurs.
- Connect the decision to the review artifact: the relevant artifact is a SRE review.
- Prefer evidence over preference. The best answer is the one whose behavior can be inspected, tested, measured, or safely changed.

##### Practical Guidance

- Write a one-sentence problem statement for `Production Readiness Review` before changing code, architecture, or process.
- Identify the owner of the decision and the people who will maintain, operate, or integrate with it.
- Name the boundary: inputs, outputs, collaborators, data, trust level, lifecycle, and failure behavior.
- Choose the smallest mechanism that preserves the invariant while keeping the design understandable.
- Add an explicit review signal: test, metric, trace, decision record, schema check, benchmark, or runbook step.
- Document the trade-off in the place future maintainers will actually see during review or operation.
- Revisit the decision when scale, ownership, product behavior, compliance pressure, or failure evidence changes.

##### Trade-offs

- Specific guidance for `Production Readiness Review` improves consistency, but applied mechanically it can ignore local constraints.
- Extra abstraction can isolate variation, but it also adds names, lifecycle, and debugging paths that must be owned.
- More validation and documentation reduce ambiguity, but they should target real risk rather than slowing harmless change.
- A local improvement can increase cross-team coordination cost if the boundary or ownership model is unclear.
- The right level of rigor depends on impact: high-risk work deserves stronger evidence than a small internal cleanup.

##### Examples

A failure-mode note makes the design operational:

```text
Dependency or component: what can fail independently.
Timeout: the maximum time callers wait.
Retry rule: when retry is safe and when it amplifies load.
Fallback: degraded behavior that protects users or operators.
Signal: metric, trace, log, or alert that proves the path occurred.
```

For `Production Readiness Review`, the value is that the system has a defined behavior when the happy path disappears.

##### Common Mistakes

- tracking vanity metrics
- writing runbooks nobody rehearses
- treating every alert as equally urgent
- Optimizing the first implementation while ignoring migration, rollback, observability, and long-term ownership.
- Reviewing surface shape instead of the invariant, failure behavior, and change cost the section is meant to control.

##### Review Checklist

- [ ] The reason to apply `Production Readiness Review` is clear in the current `Reliability Programs` context.
- [ ] The invariant is stated: the system should have a known user impact, owner, signal, and recovery path for important failures.
- [ ] The design names owners, boundaries, collaborators, and failure behavior.

- [ ] The trade-off is explicit enough that a future maintainer can revisit it.
- [ ] The solution is no more complex than the risk or variation requires.
