#### 3.1.4 Builder

##### Purpose

`Builder` belongs to the `Creational Patterns` chapter of `Design Patterns`. In this context, it is about separating construction steps from the final object so complex objects are assembled safely and readably. The practical goal is to help engineers make a decision that can survive code review, design review, production operation, and later change.

This topic matters because `Design Patterns` is concerned with using proven design shapes only when they clarify collaboration, variation, or lifecycle. A good treatment of `Builder` should clarify intent, participants, variation point, and misuse risk, protect the invariant that an object is not created until required fields and consistency checks are satisfied, and give the team observable signals such as simpler clients, explicit roles, and removable indirection.

##### Core Ideas

- Start with intent. State why `Builder` is relevant to the current problem before naming a mechanism.
- Use the chapter boundary. `Creational Patterns` supplies the nearby vocabulary, constraints, and failure modes.
- Protect the invariant: an object is not created until required fields and consistency checks are satisfied.
- Use this topic when valid construction requires several optional or ordered decisions.
- Connect the decision to the review artifact: the relevant artifact is a pattern review.
- Prefer evidence over preference. The best answer is the one whose behavior can be inspected, tested, measured, or safely changed.

##### Practical Guidance

- Write a one-sentence problem statement for `Builder` before changing code, architecture, or process.
- Identify the owner of the decision and the people who will maintain, operate, or integrate with it.
- Name the boundary: inputs, outputs, collaborators, data, trust level, lifecycle, and failure behavior.
- Choose the smallest mechanism that preserves the invariant while keeping the design understandable.
- Add an explicit review signal: test, metric, trace, decision record, schema check, benchmark, or runbook step.
- Document the trade-off in the place future maintainers will actually see during review or operation.
- Revisit the decision when scale, ownership, product behavior, compliance pressure, or failure evidence changes.

##### Trade-offs

- Specific guidance for `Builder` improves consistency, but applied mechanically it can ignore local constraints.
- Extra abstraction can isolate variation, but it also adds names, lifecycle, and debugging paths that must be owned.
- More validation and documentation reduce ambiguity, but they should target real risk rather than slowing harmless change.
- A local improvement can increase cross-team coordination cost if the boundary or ownership model is unclear.
- The right level of rigor depends on impact: high-risk work deserves stronger evidence than a small internal cleanup.

##### Examples

A useful pattern note should name the participants and the reason for the indirection:

```text
Problem: direct clients would need to know too much about construction, structure, or collaboration.
Pattern intent: separating construction steps from the final object so complex objects are assembled safely and readably.
Use when: valid construction requires several optional or ordered decisions.
Protected invariant: an object is not created until required fields and consistency checks are satisfied.
Review focus: remove the pattern if it no longer simplifies clients or isolates variation.
```

For `Builder`, a concrete application is: A query builder can require a source and projection before `build()` creates an immutable query plan. The important review question is whether the pattern makes the object collaboration easier to understand, or merely adds names around code that was already simple.

##### Common Mistakes

- using builders for trivial objects
- allowing invalid intermediate state to escape
- duplicating validation in many builders
- Optimizing the first implementation while ignoring migration, rollback, observability, and long-term ownership.
- Reviewing surface shape instead of the invariant, failure behavior, and change cost the section is meant to control.

##### Review Checklist

- [ ] The reason to apply `Builder` is clear in the current `Creational Patterns` context.
- [ ] The invariant is stated: an object is not created until required fields and consistency checks are satisfied.
- [ ] The design names owners, boundaries, collaborators, and failure behavior.
- [ ] Evidence exists through simpler clients, explicit roles, and removable indirection.
- [ ] The trade-off is explicit enough that a future maintainer can revisit it.
- [ ] The solution is no more complex than the risk or variation requires.
