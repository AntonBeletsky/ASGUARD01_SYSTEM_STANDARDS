#### 3.1.5 Prototype

##### Purpose

`Prototype` belongs to the `Creational Patterns` chapter of `Design Patterns`. In this context, it is about creating new objects from existing configured instances when copying is clearer than rebuilding. The practical goal is to help engineers make a decision that can survive code review, design review, production operation, and later change.

This topic matters because `Design Patterns` is concerned with using proven design shapes only when they clarify collaboration, variation, or lifecycle. A good treatment of `Prototype` should clarify intent, participants, variation point, and misuse risk, protect the invariant that copied instances must not accidentally share mutable state that should be independent, and give the team observable signals such as simpler clients, explicit roles, and removable indirection.

##### Core Ideas

- Start with intent. State why `Prototype` is relevant to the current problem before naming a mechanism.
- Use the chapter boundary. `Creational Patterns` supplies the nearby vocabulary, constraints, and failure modes.
- Protect the invariant: copied instances must not accidentally share mutable state that should be independent.
- Use this topic when initialization is expensive or many objects share a mostly identical configuration.
- Connect the decision to the review artifact: the relevant artifact is a pattern review.
- Prefer evidence over preference. The best answer is the one whose behavior can be inspected, tested, measured, or safely changed.

##### Practical Guidance

- Write a one-sentence problem statement for `Prototype` before changing code, architecture, or process.
- Identify the owner of the decision and the people who will maintain, operate, or integrate with it.
- Name the boundary: inputs, outputs, collaborators, data, trust level, lifecycle, and failure behavior.
- Choose the smallest mechanism that preserves the invariant while keeping the design understandable.
- Add an explicit review signal: test, metric, trace, decision record, schema check, benchmark, or runbook step.
- Document the trade-off in the place future maintainers will actually see during review or operation.
- Revisit the decision when scale, ownership, product behavior, compliance pressure, or failure evidence changes.

##### Trade-offs

- Specific guidance for `Prototype` improves consistency, but applied mechanically it can ignore local constraints.
- Extra abstraction can isolate variation, but it also adds names, lifecycle, and debugging paths that must be owned.
- More validation and documentation reduce ambiguity, but they should target real risk rather than slowing harmless change.
- A local improvement can increase cross-team coordination cost if the boundary or ownership model is unclear.
- The right level of rigor depends on impact: high-risk work deserves stronger evidence than a small internal cleanup.

##### Examples

A useful pattern note should name the participants and the reason for the indirection:

```text
Problem: direct clients would need to know too much about construction, structure, or collaboration.
Pattern intent: creating new objects from existing configured instances when copying is clearer than rebuilding.
Use when: initialization is expensive or many objects share a mostly identical configuration.
Protected invariant: copied instances must not accidentally share mutable state that should be independent.
Review focus: remove the pattern if it no longer simplifies clients or isolates variation.
```

For `Prototype`, a concrete application is: A report template can be cloned with a new date range while keeping chart definitions and access policy unchanged. The important review question is whether the pattern makes the object collaboration easier to understand, or merely adds names around code that was already simple.

##### Common Mistakes

- shallow-copying nested mutable data
- hiding expensive copy behavior
- using prototypes where explicit constructors would be clearer
- Optimizing the first implementation while ignoring migration, rollback, observability, and long-term ownership.
- Reviewing surface shape instead of the invariant, failure behavior, and change cost the section is meant to control.
